# src/daemon/

Unix socket server layer managing session lifecycle, tmux panes, and Claude orchestrator/agent spawning.

## Module Responsibilities

- **index.ts** ‚Äî Daemon entry point: acquires PID lock, starts server + monitor, recovers active/paused sessions on startup with **stuck-session detection**, handles graceful shutdown (SIGTERM/SIGINT), exposes stop/restart CLI commands. **`restart` branches on launchd**: if `launchctl list com.sisyphus.daemon` succeeds, stops the process and polls up to 3s for launchd to respawn (then exits) ‚Äî does NOT start in-process; falls back to in-process start only when no launchd service found. `acquirePidLock()` exits with code 0 (not 1) when daemon is already running.
- **server.ts** ‚Äî Listens on `~/.sisyphus/daemon.sock`, parses JSON line-delimited requests, routes to SessionManager. Maintains in-memory session tracking maps (cwd, tmux, window), persists registry to disk for recovery. Tracks message counters per session for `message` protocol requests.
- **session-manager.ts** ‚Äî Entry point for all session operations. **Lifecycle**: `startSession()` (create + spawn orchestrator, fire-and-forget Haiku naming with pane label updates ‚Äî retries up to 5 suffixed variants on collision, gives up silently if all taken), `resumeSession()` (reuse or recreate tmux, mark lost agents), `reopenWindow()` (recreates if killed), `pruneOldSessions()` (keeps 10 recent or 7-day sessions). **Handlers**: `handleSpawn()` (re-activates `completed` sessions before spawning ‚Äî allows orchestrator to spawn agents after a session is marked completed mid-cycle), `handleSubmit()`/`handleReport()`, `handleYield()` (re-activates `paused` sessions before proceeding, then guards against pane monitor races), `handleComplete()` (flushes timers + switches to home session), `handleKill()` (atomic cleanup), `handleRestartAgent()`/`handleKillAgent()`, `handleRollback()` (validates cycle, kills running agents, restores snapshot, cleanup), `handlePaneExited()` (unexpected exits ‚Üí respawn or notifications). **Respawn coordination**: `onAllAgentsDone()` uses `pendingRespawns` + `orchestratorDone` guards to prevent concurrent respawns; creates cycle snapshot, ensures window exists, respawns orchestrator via `setImmediate`.
- **orchestrator.ts** ‚Äî Spawns/respawns orchestrator Claude each cycle with mode-specific system prompts (base + strategy/implementation/validation/completion suffixes; strategy default). Injects available agent types, caches window/pane IDs, writes prompts to `prompts/` subdir. Formats session state: goal, context, messages, cycle history (agent reports + status), repositories (branches, dirty flags, agent assignments). `detectRepos()` scans for git repos and filters by config.repos if present.
- **agent.ts** ‚Äî Spawns agent Claude instances (Anthropic Claude or OpenAI Codex per provider); auto-increments per-session counter. Resolves agent type configs via frontmatter (model, color, effort, interactive). Creates Claude plugins with conditional hooks: `Stop` hook required for non-interactive agents; agent-type-specific `UserPromptSubmit` hooks for (plan, review, review-plan, debug, operator, test-spec, explore). Supports sub-agent definitions: copies subdirectories alongside agent type files. Handles agent restart and report submission.
- **summarize.ts** ‚Äî Optional async utilities via Haiku SDK: `generateSessionName()` creates kebab-case session names from task descriptions (called by session-manager on creation). `summarizeReport()` condenses agent reports to one-sentence summaries. Both fire-and-forget with 5-minute cooldown on auth failures.
- **frontmatter.ts** ‚Äî Parses YAML frontmatter from agent type definitions. Extracts metadata (name, model, color, skills, permissionMode, effort, interactive, description). Detects provider (Anthropic vs. OpenAI) from model. Resolves agent type paths (project ‚Üí user ‚Üí bundled ‚Üí installed plugins) via `resolveAgentTypePath()`. Discovers available agent types via `discoverAgentTypes()`.
- **pane-monitor.ts** ‚Äî Background poller (5s interval, sleep-aware elapsed time calculation). **Pane liveness**: detects unexpected pane kills, stuck sessions (all agents done + no orchestrator), marks running agents as lost; invokes respawn callback via `onAllAgentsDone()`. **Active time tracking**: maintains in-memory timers (`activeTimers` map) for sessions/agents/cycles; `initTimers()` at session start, `pollSession()` accumulates milliseconds, `flushTimers()` persists deltas to state. Tracks sessions with `trackSession/untrackSession` lifecycle.
- **pane-registry.ts** ‚Äî Central registry mapping paneId ‚Üí {sessionId, role, agentId}. Enables fast pane lookup and exit notification routing.
- **respawn-guard.ts** ‚Äî Tracks sessions in yield‚Üírespawn transition via `respawningSessions` set to prevent pane monitor from pausing the session while orchestrator is being respawned.
- **notify.ts** ‚Äî Desktop notifications via a **long-lived singleton process** (`SisyphusNotify.app`) that accepts JSON lines on stdin ‚Äî not a per-notification subprocess. Three-tier fallback: native app ‚Üí `terminal-notifier` ‚Üí `osascript`. Native app must be installed separately at `~/.sisyphus/SisyphusNotify.app/` (not bundled). Click-to-switch is **iTerm2-only** (AppleScript); switch script written lazily to `~/.sisyphus/notify-switch.sh`. tmux socket hardcoded to `/tmp/tmux-{uid}/default` ‚Äî fails with non-default socket paths. Switch script hardcodes `/opt/homebrew/bin/tmux` ‚Äî silently fails on Intel Macs (Homebrew at `/usr/local/bin`).
- **status-dots.ts** ‚Äî Computes per-session phase indicators for a tmux dashboard window. Phase detection reads Claude hook state files from `/tmp/claude-tmux-state/{paneNumericId}` (written by Claude Code hooks, not sisyphus). Both `stopped` and `idle` map to `orchestrator:idle` (needs input) ‚Äî only `processing` (or missing file) maps to `orchestrator:processing`. **`between-cycles` phase has two triggers**: (1) `respawningSessions.has(sessionId)` (from respawn-guard.ts) ‚Äî the expected yield‚Üírespawn window; (2) fallback when orchestrator pane is gone and no agents are running but session is still `active` ‚Äî this is the "limbo" state after an unexpected orchestrator exit before pane-monitor fires `onAllAgentsDone()`. `recomputeDots()` **groups sessions by cwd** ‚Äî finds the dashboard window via `@sisyphus_dashboard` option on the home session (30s cache), then writes `@sisyphus_dots` (a raw tmux format string `#[fg=color]icon`) to that window; also writes `@sisyphus_phase` per session. Exports `DOT_MAP` (phase‚Üíicon/color) and `getSisyphusPhases()` (populated each cycle) for `status-bar.ts`. `markSessionCompleted()` adds a 5-min TTL entry so completed dots persist after the session leaves active tracking ‚Äî called on `handleComplete()` only, not `handleKill()`. `setTrackedEntriesProvider()` must be called at startup or `recomputeDots()` is a no-op.
- **status-bar.ts** ‚Äî Renders the complete tmux status bar string and writes it to global option `@sisyphus_status`. Called after `recomputeDots()` on each poll cycle and once on startup. Pipeline: `listAllPanes()` (one tmux exec) ‚Üí read Claude state files per pane ‚Üí compute highest-priority state per session (processing > stopped > idle) ‚Üí separate normal vs sisyphus sessions (`ssyph_` prefix) ‚Üí order by `~/.config/tmux/session-order` (30s cache) ‚Üí render tmux format strings with per-client `#{session_name}` conditionals for current-session highlighting ‚Üí `setGlobalOption`. 3 tmux execs total (list-panes -a, list-sessions, set-option -g). Normal sessions show Claude activity dot; sisyphus sessions show phase icon from `DOT_MAP`.
- **state.ts** ‚Äî Session creation, mutation, and persistence. Uses **session-level mutex** (`withSessionLock`) to prevent read-modify-write races during concurrent updates. Atomically writes state via temp file + rename. **Manages cycle snapshots**: `createSnapshot()` (at cycle boundaries), `restoreSnapshot()` (restore to prior state), `listSnapshots()` (available snapshots), `deleteSnapshotsAfter()` (cleanup post-rollback).
- **tmux.ts** ‚Äî tmux CLI wrapper: session/window/pane creation and lifecycle. **Session defaults**: every `createSession()` calls `configureSessionDefaults()` which sets `pane-border-status top`, disables `allow-rename`/`automatic-rename` (prevents tmux from clobbering pane titles we set), and installs `after-kill-pane`/`pane-exited` hooks that auto-retile to `even-horizontal`. **Also renames the window to `"main"`** after disabling automatic-rename ‚Äî necessary because `automatic-rename` may have already changed the window name to the cwd basename before we disable it; if another session has a window with that same basename, `tmux set-option -t "session:basename"` silently targets the wrong session. **Pane creation**: `createPane()` targets the first/last existing pane (not window) to guarantee absolute left/right placement ‚Äî using the window as target splits relative to the active pane. **Home session support**: `findHomeSession()` locates non-sisyphus sessions in same cwd (skips `ssyph_`-prefixed names); `switchAttachedClients()` moves attached clients to home session before destroying sisyphus session. **Global helpers**: `setGlobalOption()`, `listAllSessions()`, `listAllPanes()` ‚Äî used by `status-bar.ts` for whole-server status rendering. **Pane styling**: `setPaneStyle()` stores metadata as per-pane user vars (`@pane_role`, `@pane_session`, `@pane_cycle`, `@pane_mode`, `@pane_color`) and `pane-border-format` ‚Äî all set with `-p` (per-pane flag) so each pane has its own label format; window-level `pane-border-style` uses `#{@pane_color}` which resolves per-pane at render time (last-write-wins for the window option itself). `updatePaneMeta()` updates individual vars without rebuilding the full format string (used by auto-naming).
- **colors.ts** ‚Äî Color cycling for panes: orchestrator always yellow, agents rotate deterministically.

## Key Patterns

**Respawn guard race prevention**: `respawningSessions` tracks sessions in yield‚Üírespawn transition. Pane monitor skips empty-window detection for guarded sessions (window may be temporarily empty between killing orchestrator and spawning new one). Guard added on yield/respawn entry, cleared on completion or error ‚Äî but **not cleared if `onAllAgentsDone()` returns early** (e.g., session not `active` at entry). An early-return without clearing leaves the session stuck in `between-cycles` forever.

**Active time accumulation**: Pane monitor maintains in-memory timers for each session (session, agents, cycles). On each poll, increments timers by elapsed time if panes are alive (using sleep-aware calculation). `flushTimers()` persists deltas to state file. Called before session completion/kill/rollback.

**Session completion & client switching**: When completing or killing a session via `handleComplete()`/`handleKill()`, `switchToHomeSession()` moves any attached tmux clients to a home session (if available in same cwd) before destroying the sisyphus session. Prevents users from being left in a killed session.

**Session creation**: Creates tmux session internally with deterministic naming (`sisyphus-{name}` if provided, else `sisyphus-{sessionId[:8]}`). Validates custom names against alphanumeric/hyphen/underscore pattern. Auto-generates human-readable session names from task via Haiku (fire-and-forget, failures are silent). Kills initial pane spawned by `tmux new-session` after orchestrator spawns its own.

**Agent spawning**: Increments counter, resolves agent type frontmatter (model, color, effort, interactive), detects provider (Anthropic Claude or OpenAI Codex), creates Claude plugin with conditional hooks per agent type, spawns pane.

**Session resumption**: Reuses existing tmux session if available, else creates fresh one. Uses `listPanes()` to detect live agent panes and mark lost agents. Reuses windowId from state if tmux session still exists. Kills initial pane after orchestrator spawns.

**Window reopening**: `reopenWindow()` recreates tmux window/session if killed (e.g., after orchestrator yields with no running agents), reuses existing IDs if available.

**Respawn on all-agents-done**: When all agents finish, `onAllAgentsDone()` adds session to `pendingRespawns`, creates snapshot of current cycle state, ensures tmux session/window exist (recreates if necessary), then respawns orchestrator via `setImmediate`. Uses `pendingRespawns` + `orchestratorDone` guards to prevent concurrent respawns. `orchestratorDone` is checked both in-memory AND by reading `lastCycle.completedAt` from persisted state ‚Äî the state fallback handles daemon restart between orchestrator yield and agents finishing. `respawningSessions` guard (via respawn-guard.ts) prevents pane monitor from pausing during yield‚Üírespawn window.

**Session recovery on startup**: Registry loads; `active`/`paused` sessions recover. Detects stuck sessions (all agents done, orchestrator pane gone) and immediately triggers respawn.

**Cycle snapshots**: At cycle boundaries, `state.createSnapshot()` captures full session state. `handleRollback()` restores to prior snapshot, killing running agents, removing post-rollback snapshots. `handleRollback()` does **not** restart the session ‚Äî leaves it in restored state; caller must issue `resumeSession()` explicitly.

**Orchestrator crash branching**: `handlePaneExited(role='orchestrator')` forks on agent count: if agents are running ‚Üí set guard + wait; if all agents already done ‚Üí trigger `onAllAgentsDone()`; if `agents.length === 0` ‚Üí pause session (no respawn target). Completed agent panes are killed **lazily** inside `onAllAgentsDone`'s `setImmediate` callback after the new orchestrator spawns, not at submit time.

**Kill-agent ordering**: `handleKillAgent()` unregisters the pane from `pane-registry` before killing the tmux pane. This prevents pane monitor from seeing an unexpected exit and triggering an unintended respawn for a user-initiated kill.

**Prompts**: System + user prompts written to `prompts/` subdir. Passed via Claude flags to avoid tmux quoting issues.

## Constraints

- Never write state directly ‚Äî use `state.ts` atomic writes
- Always unregister panes from pane-registry and pane-monitor when killing/completing
- Prompt files must be on disk in `prompts/` subdir
- tmux errors are fatal ‚Äî propagate, don't swallow
- Session recovery must detect and recover stuck sessions immediately on startup
- Agent type frontmatter must be valid YAML; gracefully fall back if missing or malformed
- Snapshots created at cycle boundaries; rollback validates snapshot existence
- Kill initial pane after orchestrator spawns (tmux creates it but orchestrator needs its own)
- Custom session names must be alphanumeric, hyphens, underscores only
- Interactive agents (interactive: true) skip the Stop hook requirement; all others require submit hook
- Haiku-based naming/summarization is optional (fire-and-forget); auth failures silence feature temporarily with 5-minute cooldown
- Active time must be flushed to state before session completion/kill/rollback (prevents loss of accumulated time data)
- Respawn guard must be cleared on respawn completion or error (otherwise pane monitor stays stuck); `onAllAgentsDone()` early-return paths do not clear it automatically
- `orchestratorDone` is NOT cleared on session pause/resume ‚Äî if a session is paused after orchestrator yields and before agents finish, `onAllAgentsDone()` will return early (non-active status) and neither clears the guard nor respawns; subsequent `handleYield()` after resume finds `orchestratorDone` set and skips respawn silently. Recovery: kill + restart the session.
- Call `recomputeDots()` (best-effort try/catch) after any handler that changes session phase ‚Äî omitting it leaves the dashboard stale until the next pane-monitor poll
