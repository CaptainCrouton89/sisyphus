# src/daemon/

Unix socket server layer managing session lifecycle, tmux panes, and Claude orchestrator/agent spawning.

## Module Responsibilities

- **index.ts** ‚Äî Daemon entry point: acquires PID lock, starts server + monitor, recovers active/paused sessions on startup with **stuck-session detection**, handles graceful shutdown (SIGTERM/SIGINT), exposes stop/restart CLI commands. **Node.js v22+ is enforced at the top of the file before any imports** ‚Äî exits with code 1 if the version is below 22; this check runs before config or anything else loads. **All `console.log`/`console.error` calls are patched at startup** to prepend ISO timestamps ‚Äî affects every downstream module that logs to console; do not assume bare log output in daemon.log. **`restart` branches on launchd**: if `launchctl list com.sisyphus.daemon` succeeds, stops the process and polls up to 3s for launchd to respawn (then exits) ‚Äî does NOT start in-process; falls back to in-process start only when no launchd service found. `acquirePidLock()` exits with code 0 (not 1) when daemon is already running. **`checkAndApply()` (auto-update) runs before `acquirePidLock()`** ‚Äî two simultaneous daemon starts (e.g., launchd respawn racing a manual start) can both enter the update path and attempt `npm install -g sisyphi` concurrently; npm's own locking usually prevents corruption but is not guaranteed; dev installs with symlink detection are exempt. **Startup initialization ordering**: `setRespawnCallback()`, `setTrackedEntriesProvider()`, and `setDotsCallback()` must all be called before `startMonitor()` ‚Äî first poll cycle silently has no respawn handler or dots writer if any are missing; `startServer()` must complete before `recoverSessions()` because recovery calls server-side registration functions. **`recoverSessions()` multi-stage tmux re-binding**: (1) `sessionExistsById(currentTmuxId)` checked first; (2) if gone, tries `sessionNameTaken(session.tmuxSessionName)` ‚Äî name may now belong to a different tmux session post-restart; (3) calls `resolveSessionId()` to re-capture the ID; if re-resolution returns null despite name match, `sessionAlive` is false and session is silently paused rather than recovered. **`recoverSessions()` window ID fallback**: if `tmuxWindowId` is missing from persisted state (e.g., crash between `registerSessionCwd()` and `registerSessionTmux()`), calls `getFirstWindowId()` by ID then by name ‚Äî if found, writes the discovered windowId back to state and re-calls `initSessionMeta()` to re-sync tmux session options. **Orchestrator recovery during `recoverSessions()` scans the last incomplete cycle for a non-null `paneId`** ‚Äî if the orchestrator had yielded but `paneId` was never written (e.g., crash before state flush), the cycle is skipped and the session recovers without a live orchestrator; this leaves the session `active` but stalled until a manual `sisyphus rollback` or re-yield.
- **server.ts** ‚Äî Listens on `~/.sisyphus/daemon.sock`, parses JSON line-delimited requests, routes to SessionManager. Maintains in-memory session tracking maps (cwd, tmux, window), persists registry to disk for recovery. **`persistSessionRegistry()` only writes `cwd`** ‚Äî tmux/window metadata is in-memory only. If the daemon crashes after `registerSessionCwd()` but before `registerSessionTmux()`, the recovered session has cwd but no tmux binding. **`messageCounter` is in-memory, initialized to 0 for all sessions (including resumed ones)** ‚Äî on daemon restart, IDs restart from `msg-001`, potentially colliding with IDs already written to disk. **`status` request overlays live pane-monitor timer values** (via `getActiveTimers()`) onto the persisted session before returning ‚Äî reported `activeMs` for session, agents, and cycles may exceed last-flushed persisted values by up to one poll interval; callers cannot distinguish persisted vs live. `companion` protocol request reads/updates companion name and returns full `CompanionState` ‚Äî the only companion field mutable via protocol (name). **`pane-exited` handler unregisters the pane before calling `handlePaneExited()`** ‚Äî if the handler throws, the pane is gone from the registry but session state is never updated; subsequent `pane-exited` events for the same pane return `{ok:true}` with no effect. **Disk-recovery pattern**: `resume`, `rollback`, `reopen-window`, and `reconnect` all fall back to loading from disk when the session is not in the in-memory map ‚Äî each checks `{cwd}/.sisyphus/sessions/{sessionId}/state.json`; `rollback` calls `registerSessionCwd()` (re-persists to registry), while `resume` and `reopen-window` construct a minimal tracking entry without registering. **`reconnect` protocol** re-binds a live tmux session to the daemon after a restart: calls `sessionManager.reconnectSession()` which finds the session by name, re-captures ID and windowId, and updates all in-memory maps ‚Äî use when daemon restarted mid-session and the tmux session is still running but the registry is stale. **`delete` protocol** kills the session if it's active (kill errors are swallowed ‚Äî unlike the `kill` handler which propagates), removes the entry from the registry, then removes the entire session directory with `rmSync({ recursive: true, force: true })` ‚Äî irreversible, no confirmation.
- **session-manager.ts** ‚Äî Entry point for all session operations. **Lifecycle**: `startSession()` (create + spawn orchestrator, fire-and-forget Haiku naming with pane label updates ‚Äî retries up to 5 suffixed variants on collision, gives up silently if all taken), `resumeSession()` (reuse or recreate tmux, mark lost agents ‚Äî **lost-agent detection is conditional on `session.tmuxWindowId` being non-null**: if windowId is missing, e.g., daemon crashed between `registerSessionCwd()` and `registerSessionTmux()`, all agents remain `running` in state but are unreachable with no recovery path short of `handleKill()`), `reopenWindow()` (recreates if killed), `pruneOldSessions()` (keeps 10 recent or 7-day sessions). **`reconnectSession(sessionId, cwd)`** re-binds a daemon-untracked session to a live tmux session by name ‚Äî unlike `resumeSession()` which may create a new tmux session, `reconnectSession()` throws if the tmux session doesn't exist by name (directs user to `sisyphus resume`); it re-captures `$N` ID and windowId, calls `initSessionMeta()`, then updates `registerSessionTmux()` / `trackSession()` / `updateTrackedWindow()` in order. **Handlers**: `handleSpawn()` (re-activates `completed` sessions before spawning ‚Äî allows orchestrator to spawn agents after a session is marked completed mid-cycle), `handleSubmit()`/`handleReport()`, `handleYield()` (re-activates `paused` sessions before proceeding, then guards against pane monitor races), `handleComplete()` (flushes timers + **full teardown**: untrack, unregister panes, clear agent counter, delete orchestratorDone, switch to home session, kill tmux session ‚Äî mirrors `handleKill()`; add any new complete path here too or it leaks), `handleKill()` (atomic cleanup), `handleContinue()` (re-activates a `completed` session ‚Äî delegates to `state.continueSession()`; the only protocol path that transitions out of `completed` without spawning an agent), `handleRestartAgent()`/`handleKillAgent()`, `handleRollback()` (validates cycle, kills running agents, restores snapshot, cleanup), `handlePaneExited()` (unexpected exits ‚Üí respawn or notifications). **Respawn coordination**: `onAllAgentsDone()` uses `pendingRespawns` + `orchestratorDone` guards to prevent concurrent respawns; creates cycle snapshot, ensures window exists, respawns orchestrator via `setImmediate`. Inside `setImmediate`: if session is `paused` AND still in `respawningSessions` (pane monitor raced during yield‚Üírespawn window), it's re-activated then respawned; if `paused` WITHOUT the guard, the guard is cleared and respawn is skipped (legitimate pause, not a race). **`pendingRespawns` has no TTL** ‚Äî if the `setImmediate` callback throws after `pendingRespawns.add()` but before `pendingRespawns.delete()` (e.g., snapshot creation failure), the session is permanently blocked from all future respawns; recovery requires `handleKill()` + `startSession()`. **Startup double-trigger**: `recoverSessions()` calls `onAllAgentsDone()` immediately for stuck sessions, and the first pane-monitor poll cycle independently detects them as stuck and calls it again; `pendingRespawns` guard absorbs the second call ‚Äî no double respawn ‚Äî but be aware that any logging or metrics around `onAllAgentsDone` entries will fire twice. **`pruneOldSessions()` silently skips directories with unreadable state files** ‚Äî corrupted session dirs (state.json missing or invalid JSON) are excluded from the deletion candidates and accumulate on disk without warning. **Companion hooks**: fire at `startSession` (`onSessionStart`), `handleSpawn` (`onAgentSpawned` + async `generateNickname` written back via `state.updateAgent` ‚Äî if the session completes before Haiku responds, the `updateAgent` call silently discards), `handleComplete` (`onSessionComplete` ‚Äî detects level-up + achievements, each triggers an independent async `generateCommentary` + `showCommentaryPopup` chain; also fires `generateSentiment` fire-and-forget which may call `writeSessionSummary` a second time to add the `sentiment` field after Haiku responds), `handlePaneExited` (`onAgentCrashed`). All hooks are fire-and-forget in try/catch; errors never propagate into the handler. **`fireCommentary()` helper**: the `companion` arg passed to `generateCommentary` is prompt context only ‚Äî the `.then()` callback calls `loadCompanion()` fresh before saving `lastCommentary`, so saves never stomp in-flight state mutations. `flash=true` triggers `showCommentaryPopup()` (see `companion-popup.ts`) for `session-complete`, `level-up`, and `achievement`; `session-start` and `cycle-boundary` fire commentary without a popup; `agent-crash` returns null from `shouldGenerateCommentary()` ‚Äî no commentary fires.
- **orchestrator.ts** ‚Äî Spawns/respawns orchestrator Claude each cycle with mode-specific system prompts (base + strategy/implementation/validation/completion suffixes; strategy default). Injects available agent types, caches window/pane IDs, writes prompts to `prompts/` subdir. Formats session state: goal, context, messages, cycle history (agent reports + status), repositories (branches, dirty flags, agent assignments). `detectRepos()` scans for git repos and filters by config.repos if present. **`modeContentBuilders` map** (`Record<string, ModeContentBuilder>`) extends the *user* prompt (not system prompt) with mode-specific content ‚Äî currently only `completion` is registered. `buildCompletionContent()` appends an agent summary table (id/name/type/status/final-report-summary), inline `cycle-*.md` log files from `logsDir`, and a reference path to `reportsDir` ‚Äî all read at prompt-build time, so missing dirs/files are silently skipped. To add content for a new mode, add a key to `modeContentBuilders`; the base state formatting is shared across all modes.
- **agent.ts** ‚Äî Spawns agent Claude instances (Anthropic Claude or OpenAI Codex per provider); auto-increments per-session counter. Resolves agent type configs via frontmatter (model, color, effort, interactive). Creates Claude plugins with conditional hooks: `Stop` hook required for non-interactive agents; agent-type-specific `UserPromptSubmit` hooks for (plan, review, review-plan, debug, operator, test-spec, explore). Supports sub-agent definitions: copies subdirectories alongside agent type files. Handles agent restart and report submission. **`allAgentsDone()` checks ALL running agents in the session** ‚Äî agents can call `sisyphus spawn` to create sub-agents, and the orchestrator won't respawn until every running agent (including agent-spawned ones) finishes; also requires `session.agents.length > 0` ‚Äî zero-agent sessions never trigger all-done. **`resolveAgentActiveMs()` falls back to wall-clock** if the pane-monitor timer is 0 (agent completed before the first poll cycle ran). **`claudeSessionId`**: Anthropic agents get a fresh `randomUUID()` per spawn passed as `--session-id`; a new UUID is generated on restart (doesn't resume the prior Claude conversation). OpenAI agents have no session ID.
- **summarize.ts** ‚Äî Optional async utilities via Haiku SDK: `generateSessionName()` creates kebab-case session names; task is **sliced to 500 chars** before sending; result is **sliced to 30 chars** (can break mid-hyphen); validated against `/^[a-zA-Z0-9_-]+$/` ‚Äî multi-word or non-ASCII Haiku output returns `null` silently, causing session-manager to exhaust all 5 collision retries and keep the UUID-based name. `summarizeReport()` slices report to **3000 chars** before sending. `generateSentiment(task, messages)` generates a one-sentence emotional read on a completed session ‚Äî task sliced to **1000 chars**, first 10 messages each sliced to **500 chars**, result sliced to **200 chars**; called fire-and-forget from `handleComplete()` and enriches the already-written session summary via a second `writeSessionSummary()` call (the `sentiment` field is nullable and absent from the first write). All three share the 5-minute cooldown on auth failures (module-level in `haiku.ts`).
- **frontmatter.ts** ‚Äî Parses YAML frontmatter from agent type definitions. Extracts metadata (name, model, color, skills, permissionMode, effort, interactive, description). Detects provider (Anthropic vs. OpenAI) from model. Resolves agent type paths (project ‚Üí user ‚Üí bundled ‚Üí installed plugins) via `resolveAgentTypePath()`. Discovers available agent types via `discoverAgentTypes()`.
- **pane-monitor.ts** ‚Äî Background poller (5s interval, sleep-aware elapsed time calculation). **Pane liveness**: detects unexpected pane kills, stuck sessions (all agents done + no orchestrator), marks running agents as lost; invokes respawn callback via `onAllAgentsDone()`. **Active time tracking**: maintains in-memory timers (`activeTimers` map) for sessions/agents/cycles; `initTimers()` called at session start AND lazily inside `pollSession()` if the entry is missing (guards against races where a session begins polling before `trackSession` triggers `initTimers`); `pollSession()` accumulates milliseconds, `flushTimers()` persists deltas to state. Tracks sessions with `trackSession/untrackSession` lifecycle. **Sleep detection**: if elapsed > 3√ó poll interval, only the poll interval is credited (not actual elapsed) ‚Äî masks machine sleep to prevent inflated totals. **`activeTimers` IS flushed on graceful shutdown** (SIGTERM/SIGINT) ‚Äî `index.ts` iterates `getTrackedSessionIds()` and calls `flushTimers()` before stopping; time is only lost on daemon crash (not clean restart). **Companion mood**: recomputed each poll cycle from `MoodSignals`; **saved only when mood changes** ‚Äî `computeMood` mutates `debugMood` on the companion object even when the mood string is unchanged, but `saveCompanion` is only called inside the `if (newMood !== companion.mood)` block; `debugMood` may be slightly stale in companion.json between mood transitions. `recentCrashes` counts agents with `status === 'crashed'` and `completedAt` within last 30 minutes from tracked active sessions. `idleStartTime` clock starts when `activeTimers` becomes empty (not when `trackedSessions` empties) ‚Äî resets when any timer is re-added. Mood recompute is **throttled to once per minute when idle** (no tracked sessions ‚Äî `trackedSessions.size === 0`), every 5s poll when active. `idleStartTime`/`idleDurationMs` uses `activeTimers.keys().length === 0` ‚Äî a different idle signal. These can diverge: if a session is untracked but leaves stale `activeTimers` entries, the throttle skips recompute while `idleDurationMs` stays 0. `justCompleted`, `justCrashed`, `justLeveledUp` use **2-minute temporal decay** via module-level timestamps (`lastCompletionTime`/`lastCrashTime`/`lastLevelUpTime`) ‚Äî session-manager calls `markEventCompletion()`/`markEventCrash()`/`markEventLevelUp()` exports to set them; pane-monitor checks `(nowMs - lastXTime) < 120_000`. `currentMaxCycleCount` is tracked via `updateCycleCount()`/`resetCycleCount()` exports and flows into `signals.cycleCount`. **`idle-wake` commentary**: fires when transitioning from idle (>1 min) to active (`timerKeys.length > 0` while `idleStartTime > 0`) ‚Äî 50% chance; reloads companion fresh before saving. **`late-night` commentary**: fires during 2‚Äì6am polls when `trackedSessions.size > 0`; throttled to once per 30min via module-level `lastLateNightCommentary` timestamp; calls `showCommentaryPopup(text)` on success. **`flushTimers()` has two silent early-return paths**: (1) session has no `activeTimers` entry ‚Äî no-op; (2) session is not in `trackedSessions` ‚Äî no-op; any time accumulated in `activeTimers` for an untracked session is permanently lost without warning. Per-poll `pollSessionCache` is populated during `pollSession()` and reused for mood signal computation ‚Äî avoids re-reading state files for sessions already polled in the same cycle. **`completed` session cleanup**: `pollSession()` has a special path for `status === 'completed'` sessions ‚Äî if the tracked orchestrator pane is gone (or no orchestrator was ever registered), calls `cleanupSessionMaps()` + `untrackSession()` and returns early; this is how completed sessions leave `trackedSessions` between `handleComplete()` teardown and the next poll cycle. Sessions that complete during the poll window are cleaned up within one poll interval, not immediately.
- **pane-registry.ts** ‚Äî Central registry mapping paneId ‚Üí {sessionId, role, agentId}. Enables fast pane lookup and exit notification routing.
- **respawn-guard.ts** ‚Äî Tracks sessions in yield‚Üírespawn transition via `respawningSessions` set to prevent pane monitor from pausing the session while orchestrator is being respawned.
- **notify.ts** ‚Äî Desktop notifications via a **long-lived singleton process** (`SisyphusNotify.app`) that accepts JSON lines on stdin ‚Äî not a per-notification subprocess. Three-tier fallback: native app ‚Üí `terminal-notifier` ‚Üí `osascript`. Native app must be installed separately at `~/.sisyphus/SisyphusNotify.app/` (not bundled). Click-to-switch is **iTerm2-only** (AppleScript); switch script written lazily to `~/.sisyphus/notify-switch.sh`. tmux socket hardcoded to `/tmp/tmux-{uid}/default` ‚Äî fails with non-default socket paths. Switch script hardcodes `/opt/homebrew/bin/tmux` ‚Äî silently fails on Intel Macs (Homebrew at `/usr/local/bin`).
- **status-dots.ts** ‚Äî Computes per-session phase indicators for a tmux dashboard window. **Phase detection derives orchestrator pane from persisted cycle state** (`OrchestratorCycle.paneId` + `!completedAt`) ‚Äî does NOT use the in-memory `sessionOrchestratorPane` map from `orchestrator.ts`, so it survives daemon restarts without recovery. Claude hook state files at `/tmp/claude-tmux-state/{paneNumericId}` (written by Claude Code hooks, not sisyphus) distinguish `orchestrator:processing` from `orchestrator:idle`. Both `stopped` and `idle` ‚Üí `orchestrator:idle`; `processing` (or missing file) ‚Üí `orchestrator:processing`. **`between-cycles` phase has two triggers**: (1) `respawningSessions.has(sessionId)` (from respawn-guard.ts) ‚Äî the expected yield‚Üírespawn window; (2) fallback when orchestrator pane is gone and no agents are running but session is still `active` ‚Äî this is the "limbo" state after an unexpected orchestrator exit before pane-monitor fires `onAllAgentsDone()`. `recomputeDots()` **groups sessions by cwd** ‚Äî finds the dashboard window via `@sisyphus_dashboard` option on the home session (30s cache), then writes `@sisyphus_dots` as a tmux format string to that window **as a window option** (`setWindowOption`). **`@sisyphus_phase` is no longer written to the tmux session option** ‚Äî `recomputeDots()` only populates the in-process `sisyphusPhases` map; external tools (tmux format strings, shell scripts) that read `#{@sisyphus_phase}` will get stale/unset values. `sessionNamesWithPhase` set was added as infrastructure for future cleanup of stale phase options ‚Äî no cleanup logic exists yet. Exports `DOT_MAP` (phase‚Üíicon/color), `getSisyphusPhases()` (populated each cycle), and `getTotalRunningAgents()` for `status-bar.ts` ‚Äî **`getTotalRunningAgents()` is owned here, not in pane-monitor**; it counts `status === 'running'` agents summed across all tracked sessions and is reset to 0 at the start of each `recomputeDots()` call (returns 0 before first cycle). `markSessionCompleted()` adds a 5-min TTL entry so completed dots persist after the session leaves active tracking ‚Äî called on `handleComplete()` only, not `handleKill()`. **If a session is continued/resumed within the 5-min TTL**, it re-enters `trackedSessions` while the TTL entry still exists ‚Äî `recomputeDots()` renders both, producing a duplicate dot for that cwd; clears itself within one TTL expiry (no explicit invalidation on resume). `setTrackedEntriesProvider()` must be called at startup or `recomputeDots()` is a no-op. **`invalidateDashboardCache(cwd)` is exported but never called** ‚Äî the 30s dashboard window cache is effectively permanent-until-daemon-restart for any cwd once set; any future code path that reassigns `@sisyphus_dashboard` must call it or dots will write to the stale window. **Completed-session dots require a live home session** with `@sisyphus_dashboard` set at render time ‚Äî if the user closed the dashboard window after `handleComplete()`, the dot silently disappears even within the 5-min TTL (no error, `getDashboardWindowId` returns null and the write is skipped).
- **status-bar.ts** ‚Äî Renders the complete tmux status bar string and writes it to global option `@sisyphus_status`. Called after `recomputeDots()` on each poll cycle and once on startup. Pipeline: `listAllPanes()` (one tmux exec) ‚Üí read Claude state files per pane ‚Üí compute highest-priority state per session (processing > stopped > idle) ‚Üí separate normal vs sisyphus sessions ‚Üí order by `~/.config/tmux/session-order` (30s cache) ‚Üí render tmux format strings with per-client `#{session_name}` conditionals for current-session highlighting ‚Üí `setGlobalOption`. 3 tmux execs total (list-panes -a, list-sessions, set-option -g). **Early-return guard**: if `listAllSessions()` returns empty, `setGlobalOption` is never called and `@sisyphus_status` remains at its previous value (bar stays stale, not blank). **Powerline section bands**: renders three background bands separated by Nerd Font left-pointing arrows (`\uE0B2` U+E0B2) in fixed order: normal sessions (SESSIONS_BG `#252629`) ‚Üí sisyphus sessions (SISYPHUS_BG `#36383e`) ‚Üí companion (COMPANION_BG `#4a4d55`). Bands are conditional ‚Äî sections with no entries are omitted entirely rather than rendered empty. **Active session highlight**: the focused session uses ACTIVE_SESSION_BG (`#3d3225`, warm amber) with ACTIVE_TEXT; inactive sessions use INACTIVE_TEXT. Both normal and sisyphus bands apply this conditional via `#{?#{==:#{session_name},...}}`. **Separator arrows are also session-conditional** (`renderSessionArrow` takes the current session name + left-neighbor name) ‚Äî inactive intra-band arrows use `fg=sectionBg` (invisible; sessions run together visually); the arrow *preceding* the active session uses `fg=ACTIVE_SESSION_BG` (amber "nose"); the arrow *following* the active session uses `bg=ACTIVE_SESSION_BG` (amber "tail" ‚Äî closes the highlight). For cross-band transitions, `renderSectionBoundary` keys on the **last session of the previous band** (`trailingName`), not the first session of the new band ‚Äî if that trailing session is active, the boundary arrow renders amber. **Requires Nerd Fonts** ‚Äî missing font renders the arrow as a box/question mark without any error or fallback. Normal sessions show Claude activity dot; **sisyphus sessions always render with label `S`** (not the tmux session name) ‚Äî if multiple sisyphus sessions are active, they are indistinguishable by label; only the phase icon and active highlight differentiate them. **Sisyphus session classification uses `getSisyphusPhases()` map as the primary signal**, not the `ssyph_` prefix ‚Äî a session in the phases map renders in the sisyphus band; a session with `ssyph_` prefix that is absent from the phases map is silently dropped from both bands (never rendered). **Only normal sessions are ordered** by the session-order file; sisyphus sessions render in `phases.values()` iteration order. **Companion rendering**: passes `maxWidth: 38` and `agentCount: getTotalRunningAgents()` to `renderCompanion()` from `src/shared/companion-render.ts`; fields are **conditional on sisyphus session count** ‚Äî `['face', 'boulder', 'verb']` when any sisyphus session is active, `['face', 'boulder', 'hobby']` when idle. `verbIndex: companion.spinnerVerbIndex` is passed so `verb` cycles through `SPINNER_VERBS` ‚Äî index is advanced once per cycle boundary in `onAllAgentsDone()` and persisted to companion.json. Companion face is agent-count-aware; text beyond 38 chars is truncated before the tmux format string is built. **tmux format comma constraint**: `#[fg=x,bold]` combined attributes silently break `#{?}` conditional expressions ‚Äî all style attributes use separate `#[fg=x]#[bold]` blocks; mixing them corrupts the conditional without error. **Companion data cache**: `loadCompanion()` is cached for 10s (`cachedCompanion` / `companionCacheTime`) ‚Äî `saveCompanion()` calls from session-manager won't be reflected in the status bar until the TTL expires. Commentary popups are handled by `companion-popup.ts` via `showCommentaryPopup()`, not via status-bar flash (flash mechanism no longer exists). **Non-atomic render**: the companion section has its own try/catch and won't blank the bar on failure; earlier pipeline stages (`listAllPanes()`, session rendering) have no try/catch ‚Äî an exception there skips `setGlobalOption` entirely, leaving `@sisyphus_status` unset (blank status bar). **`ensureStatusRightIntegration()`**: called on the first `writeStatusBar()` invocation; injects `#{E:@sisyphus_status}` into `status-right` immediately before the first `#[fg=` block (before the clock section); if no `#[fg=` found, prepends. Also bumps `status-right-length` to 250 if below that. Includes a migration step: if `status-left` still contains `@sisyphus_status` (old injection location), strips it. If `status-right` already contains `@sisyphus_status`, integration is skipped (treated as already done). **Also sets three window-tab globals on first call**: `window-status-separator ''`, `window-status-format`, and `window-status-current-format` ‚Äî Powerline window-tab styling using right-pointing arrows (`\uE0B0`, distinct from section dividers `\uE0B2`). The tab format embeds `#(~/.tmux/claude-status.sh '#{window_id}')` (external script, not bundled with sisyphus) and `#{@sisyphus_dots}` ‚Äî missing script returns empty string silently via tmux `#()`, no error or fallback. `statusIntegrated` is a module-level flag never reset at runtime ‚Äî if `status-right` is manually changed in tmux after daemon start, neither the status-right injection nor the window-tab format writes will re-fire until daemon restart. **`writeStatusBar()` has no mutual exclusion** ‚Äî called from pane-monitor's poll loop and independently at startup; concurrent invocations both call `setGlobalOption()` and the second write always wins; during a flash window, a pane-monitor call that races a startup call can overwrite the flash render with a full session bar before the 10s TTL expires.
- **state.ts** ‚Äî Session creation, mutation, and persistence. Uses **session-level mutex** (`withSessionLock`) to prevent read-modify-write races during concurrent updates. **Lock is per-sessionId only** ‚Äî operations on different sessions run concurrently; callers that assume global ordering (e.g., reading one session while writing another) are not protected. Atomically writes state via temp file + rename. **Manages cycle snapshots**: `createSnapshot()` (at cycle boundaries), `restoreSnapshot()` (restore to prior state), `listSnapshots()` (available snapshots), `deleteSnapshotsAfter()` (cleanup post-rollback). **`updateAgent()` searches in reverse** ‚Äî `.slice().reverse().find()` ‚Äî so if multiple entries share the same `agentId`, the most-recently-appended one is updated; callers that push duplicate IDs will silently update the wrong record if they add rather than mutate. **`restoreSnapshot()` clears `tmuxSessionName` and `tmuxWindowId`** from the restored state ‚Äî the existing tmux session/window is NOT killed, just orphaned; `resumeSession()` must be called afterward to re-bind the session to tmux.
- **tmux.ts** ‚Äî tmux CLI wrapper: session/window/pane creation and lifecycle. **All tmux `-t` targets use the `t()` helper** (`shellQuote()` ‚Üí single-quote wrapping) ‚Äî this is required because `$34`-style session IDs contain `$` which gets shell-expanded when passed to `/bin/sh` with double-quote wrapping; never revert to `"${target}"` double-quote style on any target string. **`resolveSessionId()` uses `list-sessions` + exact name match** (not `display-message -t "=name"`) ‚Äî the old `display-message` approach requires an attached client and silently fails in headless daemon context; if you need a session's `$N` ID from its name, use `resolveSessionId()`, do not roll your own. **`getFirstWindowId(sessionTarget)`** returns the index-0 window ID for a session target ‚Äî used by recovery to discover a windowId when state was partially lost. **Session defaults**: every `createSession()` calls `configureSessionDefaults()` which sets `pane-border-status top`, disables `allow-rename`/`automatic-rename` (prevents tmux from clobbering pane titles we set), and installs `after-kill-pane`/`pane-exited` hooks that auto-retile to `even-horizontal`. Hooks are set at **session scope** ‚Äî they fire for all windows in that session, not just the initial window. The window is named `"main"` at `new-session -n main` creation time ‚Äî `configureSessionDefaults()` does NOT rename it afterward. **Session existence API ‚Äî three functions, each with a distinct use case**: `sessionExistsById(tmuxSessionId)` uses `has-session -t "$N"` ‚Äî safe because `$N` is always `$<integer>` and tmux matches it exactly (no prefix risk); `sessionNameTaken(name)` uses `list-sessions` exact match ‚Äî use only for collision detection at creation/rename; `isSessionAlive(id, name)` dispatches to `sessionExistsById` when id is set, else `sessionNameTaken` ‚Äî the standard check for "is this session still running". Never use `has-session -t` directly with session names ‚Äî it does prefix matching and returns true for `"foo"` when only `"foo-resume"` exists. **`initSessionMeta(target, cwd, sisyphusSessionId)`** sets `@sisyphus_cwd` and `@sisyphus_session_id` on every newly created session ‚Äî required for `findHomeSession()` to find the home session and for the TUI session selector. **Pane creation**: `createPane()` targets the first/last existing pane (not window) to guarantee absolute left/right placement ‚Äî using the window as target splits relative to the active pane. **Home session support**: `findHomeSession()` matches by `@sisyphus_cwd` session option, not by name ‚Äî makes one `show-options` tmux exec per session (O(n)) on each call; a non-`ssyph_` session that hasn't set this option is invisible to it; `switchAttachedClients()` moves attached clients to home session before destroying sisyphus session. **Option helpers**: `getGlobalOption()`, `setGlobalOption()`, `setSessionOption()`, `unsetSessionOption()` all use `execSafe` ‚Äî none throw; writes to missing session targets silently no-op. `listAllSessions()`, `listAllPanes()` ‚Äî used by `status-bar.ts` for whole-server status rendering. **Pane styling**: `setPaneStyle()` stores metadata as per-pane user vars (`@pane_role`, `@pane_session`, `@pane_cycle`, `@pane_mode`, `@pane_color`) and `pane-border-format` ‚Äî all set with `-p` (per-pane flag) so each pane has its own label format; window-level `pane-border-style` uses `#{@pane_color}` which resolves per-pane at render time (last-write-wins for the window option itself). The `modeSegment` in `pane-border-format` uses `#[fg=color\,italics]` (backslash-comma) inside a `#{?}` conditional ‚Äî this is the correct tmux escape for commas within conditionals in `pane-border-format` context; do not "fix" it to a space-separated block. The status-bar.ts rule (separate `#[fg=x]#[bold]` blocks) applies to `status-right` format strings, not `pane-border-format`. `updatePaneMeta()` updates individual vars without rebuilding the full format string (used by auto-naming).
- **companion.ts** ‚Äî XP/level/mood/achievement system persisted to `~/.sisyphus/companion.json` via its own temp+rename (not `state.ts` mutex). `loadCompanion()` applies forward-compat field fills on read ‚Äî includes `spinnerVerbIndex` (defaults to 0) and `baselines: defaultBaselines()`; this fill runs on every load so adding new fields here is safe across daemon upgrades. `computeMood()` with no `signals` arg returns hour-based mood only (existential 02‚Äì06, sleepy 22‚Äì02, zen otherwise) ‚Äî signals arg required for full scoring. `computeMood()` also reads `companion.recentCompletions` directly for "flow state" scoring (last completion within 30 min + active session) ‚Äî a cross-session signal baked into the companion object, not passed via `MoodSignals`. **`computeMood()` fully uses z-scores**: calls `zScore()` for all four baseline metrics (`sessionZ`, `cycleZ`, `agentZ`, `dailyZ`) and uses them in every mood's scoring formula ‚Äî active code, not future infrastructure. `zScore()` uses cold-start defaults when `count < MIN_SAMPLES (5)`, with a 20%-of-mean stddev floor and per-metric absolute floors. **Tie-breaking**: iterates `['happy', 'grinding', 'frustrated', 'zen', 'sleepy', 'excited', 'existential']` with strict `>`; `0 > -1` fires on first iteration ‚Äî `happy` wins all-zero ties (not `grinding`, despite the initializer). `onSessionComplete()` returns an array of newly unlocked `AchievementId`s (empty if none) ‚Äî callers must check this to trigger achievement commentary. XP formula: strength√ó80 + endurance/3.6M√ó15 + wisdom√ó40 + patience√ó5. Level thresholds: base 150 XP, each level multiplies by 1.35; `getTitle()` scans `TITLE_MAP` backward ‚Äî levels 21‚Äì24, 26‚Äì29, 31+ inherit the nearest lower defined title. `computeLevelProgress()` exported for progress bar rendering: returns `{xpIntoLevel, xpForNextLevel}`. **`onSessionComplete()` is delta-safe** against continue‚Üíre-complete double-crediting: reads `session.companionCreditedCycles` and `session.companionCreditedActiveMs` (written by `session-manager.ts` at completion) and credits only the delta; missing fields default to 0 (full credit on first completion). **`computeWisdomGain()` awards 0‚Äì3 wisdom points per session**: +1 for ‚â•80% of agents completing without crash/kill/lost; +1 for ‚â•2 agents per cycle average (parallelization); +1 for ‚â•2 distinct orchestrator modes used (`mode` field on cycles). No stddev gate; `isEfficientSession()` no longer exists. **Patience uses sqrt formula**: credits `ceil(sqrt(totalCycles)) ‚àí ceil(sqrt(creditedCycles))` points (diminishing returns) plus +1 for first new `validation` mode and +1 for first new `completion` mode seen in the session. **Welford baseline accumulation**: each `onSessionComplete()` feeds `activeMs`, `totalCycles`, and `agents.length` into per-metric `RunningStats` (`sessionMs`, `cycleCount`, `agentCount`) via `welfordUpdate()`. `sessionsPerDay` uses a `pendingDayCount` accumulator finalized into Welford on the first completion of a new calendar day; gap days between `lastCountedDay` and yesterday are zero-fed. **Sessions on a day that sees no subsequent completion day are permanently stuck in `pendingDayCount` and never enter the `sessionsPerDay` distribution.** **`consecutiveEfficientSessions`** increments when `totalCycles <= 3`; resets on `totalCycles > 3` ‚Äî separate from `wisdom` stat, separate from crash tracking. **`onAgentCrashed()`** only resets `consecutiveCleanSessions`; `sessionsCrashed` is incremented once per session inside `onSessionComplete()` based on whether any agent has `status === 'crashed'`, not per individual crash event. `recentCompletions` capped at 10 entries; used by `momentum` (5 completions within 4h) and `overdrive` (6+ in one calendar day from recentCompletions). **`normalizeTask()`** keys `taskHistory` as `{cwdBasename}:{task[:100].toLowerCase().trimCollapsed}` ‚Äî tasks differing only in case or whitespace normalize to the same key (intentional); repos sharing a basename collide across projects.

All non-zero ‚Äî no event type returns null unconditionally. Time-of-day personality: chipper 06‚Äì10, professional 10‚Äì17, reflective 17‚Äì22, dry-humor 22‚Äì02, delirious 02‚Äì06. **`buildSentimentContext()`** calls `getRecentSentiments(3)` from `history.ts` and injects the last 3 session sentiments (task + sentiment string) into the commentary prompt ‚Äî the only cross-module read from `history.ts` here; produces empty string when history is absent (no error). **`generateNickname()`** takes the first whitespace-split token from Haiku's response and truncates to 20 chars ‚Äî multi-word responses silently lose everything after the first word. **`nicknameStyleGuide()`**: level ‚â• 15 unconditionally returns legendary names regardless of mood/stats, overriding all mood+stat branches. **`generateRepoNickname(repoPath, memory)`** takes a `RepoMemory` struct (from `history.ts`: visits, completions, crashes, moodAvg) and generates a 1‚Äì2 word repo nickname; truncated to 30 chars; unlike `generateNickname`, multi-word output is expected ‚Äî callers must supply the memory object, this function never loads it.
- **sessions-manifest.ts** ‚Äî Writes a manifest of all active sessions (sisyphus-managed `S` and home `H`) to disk on each poll cycle via `writeManifest()`, called from `dotsCallback` in `index.ts`. TSV has a `#ts:epoch` header line. **Home sessions classified by `@sisyphus_cwd` option** (not name prefix) ‚Äî queried by session name, not `$N` ID (to avoid `$` shell expansion in `execSync`). The manifest is the canonical way for external tools (TUI, shell scripts) to discover sessions without hitting the daemon socket.
- **colors.ts** ‚Äî Color cycling for panes: orchestrator always yellow, agents rotate deterministically.

## Key Patterns

**Respawn guard race prevention**: `respawningSessions` tracks sessions in yield‚Üírespawn transition. Pane monitor skips empty-window detection for guarded sessions (window may be temporarily empty between killing orchestrator and spawning new one). Guard added on yield/respawn entry, cleared on completion or error ‚Äî but **not cleared if `onAllAgentsDone()` returns early** (e.g., session not `active` at entry). An early-return without clearing leaves the session stuck in `between-cycles` forever.

**Active time accumulation**: Pane monitor maintains in-memory timers for each session (session, agents, cycles). On each poll, increments timers by elapsed time if panes are alive (using sleep-aware calculation). `flushTimers()` persists deltas to state file. Called before session completion/kill/rollback.

**Session completion & client switching**: When completing or killing a session via `handleComplete()`/`handleKill()`, `switchToHomeSession()` moves any attached tmux clients to a home session (if available in same cwd) before destroying the sisyphus session. Prevents users from being left in a killed session.

**Session creation**: Creates tmux session internally with deterministic naming (`sisyphus-{name}` if provided, else `sisyphus-{sessionId[:8]}`). Validates custom names against alphanumeric/hyphen/underscore pattern. Auto-generates human-readable session names from task via Haiku (fire-and-forget, failures are silent). Kills initial pane spawned by `tmux new-session` after orchestrator spawns its own.

**Agent spawning**: Increments counter, resolves agent type frontmatter (model, color, effort, interactive), detects provider (Anthropic Claude or OpenAI Codex), creates Claude plugin with conditional hooks per agent type, spawns pane.

**Session resumption**: Reuses existing tmux session if available, else creates fresh one. Uses `listPanes()` to detect live agent panes and mark lost agents. Reuses windowId from state if tmux session still exists. Kills initial pane after orchestrator spawns.

**Window reopening**: `reopenWindow()` recreates tmux window/session if killed (e.g., after orchestrator yields with no running agents), reuses existing IDs if available.

**Respawn on all-agents-done**: When all agents finish, `onAllAgentsDone()` adds session to `pendingRespawns`, creates snapshot of current cycle state, ensures tmux session/window exist (recreates if necessary), then respawns orchestrator via `setImmediate`. Uses `pendingRespawns` + `orchestratorDone` guards to prevent concurrent respawns. `orchestratorDone` is checked both in-memory AND by reading `lastCycle.completedAt` from persisted state ‚Äî the state fallback handles daemon restart between orchestrator yield and agents finishing. `respawningSessions` guard (via respawn-guard.ts) prevents pane monitor from pausing during yield‚Üírespawn window.

**Session recovery on startup**: Registry loads; `active`/`paused` sessions recover. Detects stuck sessions (all agents done, orchestrator pane gone) and immediately triggers respawn.

**Cycle snapshots**: At cycle boundaries, `state.createSnapshot()` captures full session state. `handleRollback()` restores to prior snapshot, killing running agents, removing post-rollback snapshots. `handleRollback()` does **not** restart the session ‚Äî leaves it in restored state; caller must issue `resumeSession()` explicitly.

**Orchestrator crash branching**: `handlePaneExited(role='orchestrator')` forks on agent count: if agents are running ‚Üí set guard + wait; if all agents already done ‚Üí trigger `onAllAgentsDone()`; if `agents.length === 0` ‚Üí pause session (no respawn target). Completed agent panes are killed **lazily** inside `onAllAgentsDone`'s `setImmediate` callback after the new orchestrator spawns, not at submit time.

**Kill-agent ordering**: `handleKillAgent()` unregisters the pane from `pane-registry` before killing the tmux pane. This prevents pane monitor from seeing an unexpected exit and triggering an unintended respawn for a user-initiated kill.

**Companion write races**: `companion.ts` uses its own load/save (not `state.ts` mutex), so concurrent writes are possible. Async commentary callbacks always call `loadCompanion()` fresh immediately before writing `lastCommentary` to minimize the stomp window ‚Äî never capture a companion reference across an `await`. Multiple commentary chains (session-complete + level-up + achievement) can overlap; the last writer wins on `lastCommentary`, which is acceptable since each chain writes a different event type.

**Prompts**: System + user prompts written to `prompts/` subdir. Passed via Claude flags to avoid tmux quoting issues.

## Constraints

- Never write state directly ‚Äî use `state.ts` atomic writes
- Always unregister panes from pane-registry and pane-monitor when killing/completing
- Prompt files must be on disk in `prompts/` subdir
- tmux errors are fatal ‚Äî propagate, don't swallow
- Session recovery must detect and recover stuck sessions immediately on startup
- Agent type frontmatter must be valid YAML; gracefully fall back if missing or malformed
- Snapshots created at cycle boundaries; rollback validates snapshot existence
- Kill initial pane after orchestrator spawns (tmux creates it but orchestrator needs its own)
- Custom session names must be alphanumeric, hyphens, underscores only
- Session names containing tmux format special characters (`#`, `{`, `}`) break `status-bar.ts` format strings silently ‚Äî no escaping is applied
- Interactive agents (interactive: true) skip the Stop hook requirement; all others require submit hook
- Haiku-based naming/summarization is optional (fire-and-forget); auth failures silence feature temporarily with 5-minute cooldown
- Active time must be flushed to state before session completion/kill/rollback, and is flushed automatically on graceful shutdown ‚Äî only daemon crashes lose un-flushed time
- Respawn guard must be cleared on respawn completion or error; **`onAllAgentsDone()` has two early-return paths with different guard behavior**: (1) if `orchestratorDone` is not set yet (waiting for orchestrator to yield), returns without clearing ‚Äî guard stays, session correctly shows `between-cycles`; (2) if `session.status !== 'active'`, deletes the guard before returning ‚Äî but no respawn fires, leaving the session unpaused from `between-cycles` but stuck with no orchestrator. Path (2) is the dangerous one: recover by kill + restart.
- `orchestratorDone` IS cleared by `resumeSession()`, but that doesn't rescue the stuck state from path (2) above ‚Äî `onAllAgentsDone()` already returned without respawning, and `resumeSession()` doesn't re-trigger the check. The `lastCycle.completedAt` fallback only catches clean orchestrator yields ‚Äî if the orchestrator crashed (no `completedAt` written), the fallback finds nothing and respawn stays blocked even after daemon restart; recovery is kill + restart.
- Call `recomputeDots()` (best-effort try/catch) after any handler that changes session phase ‚Äî omitting it leaves the dashboard stale until the next pane-monitor poll
- Companion writes use own `loadCompanion()`/`saveCompanion()` ‚Äî never route through `state.ts`. Async commentary callbacks must reload fresh before saving (never close over a captured companion reference across an await)
- `onSessionComplete()` return value (new achievement IDs) must be inspected ‚Äî achievement commentary only fires if the array is non-empty; the companion is already mutated when the function returns
- `trackSession()` (pane-monitor) must be called before any `updateTrackedWindow()` for that session ‚Äî throws synchronously if session isn't tracked yet; session-manager calls it at session start and resume, but any server.ts recovery path that calls `updateTrackedWindow()` directly bypasses this ordering
