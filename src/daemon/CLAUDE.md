# src/daemon/

Unix socket server layer managing session lifecycle, tmux panes, and Claude orchestrator/agent spawning.

## Module Responsibilities

- **index.ts** — Daemon entry point: acquires PID lock, starts server + monitor, recovers active/paused sessions on startup with **stuck-session detection**, handles graceful shutdown (SIGTERM/SIGINT), exposes stop/restart CLI commands. **Node.js v22+ is enforced at the top of the file before any imports** — exits with code 1 if the version is below 22; this check runs before config or anything else loads. **All `console.log`/`console.error` calls are patched at startup** to prepend ISO timestamps — affects every downstream module that logs to console; do not assume bare log output in daemon.log. **`restart` branches on launchd**: if `launchctl list com.sisyphus.daemon` succeeds, stops the process and polls up to 3s for launchd to respawn (then exits) — does NOT start in-process; falls back to in-process start only when no launchd service found. `acquirePidLock()` exits with code 0 (not 1) when daemon is already running. **`checkAndApply()` (auto-update) runs before `acquirePidLock()`** — two simultaneous daemon starts (e.g., launchd respawn racing a manual start) can both enter the update path and attempt `npm install -g sisyphi` concurrently; npm's own locking usually prevents corruption but is not guaranteed; dev installs with symlink detection are exempt. **Startup initialization ordering**: `setRespawnCallback()`, `setTrackedEntriesProvider()`, and `setDotsCallback()` must all be called before `startMonitor()` — first poll cycle silently has no respawn handler or dots writer if any are missing; `startServer()` must complete before `recoverSessions()` because recovery calls server-side registration functions. **`recoverSessions()` multi-stage tmux re-binding**: (1) `sessionExistsById(currentTmuxId)` checked first; (2) if gone, tries `sessionNameTaken(session.tmuxSessionName)` — name may now belong to a different tmux session post-restart; (3) calls `resolveSessionId()` to re-capture the ID; if re-resolution returns null despite name match, `sessionAlive` is false and session is silently paused rather than recovered. **`recoverSessions()` window ID fallback**: if `tmuxWindowId` is missing from persisted state (e.g., crash between `registerSessionCwd()` and `registerSessionTmux()`), calls `getFirstWindowId()` by ID then by name — if found, writes the discovered windowId back to state and re-calls `initSessionMeta()` to re-sync tmux session options. **Orchestrator recovery during `recoverSessions()` scans the last incomplete cycle for a non-null `paneId`** — if the orchestrator had yielded but `paneId` was never written (e.g., crash before state flush), the cycle is skipped and the session recovers without a live orchestrator; this leaves the session `active` but stalled until a manual `sisyphus rollback` or re-yield.
- **server.ts** — Listens on `~/.sisyphus/daemon.sock`, parses JSON line-delimited requests, routes to SessionManager. Maintains in-memory session tracking maps (cwd, tmux, window), persists registry to disk for recovery. **`persistSessionRegistry()` only writes `cwd`** — tmux/window metadata is in-memory only. If the daemon crashes after `registerSessionCwd()` but before `registerSessionTmux()`, the recovered session has cwd but no tmux binding. **`messageCounter` is in-memory, initialized to 0 for all sessions (including resumed ones)** — on daemon restart, IDs restart from `msg-001`, potentially colliding with IDs already written to disk. **`status` request overlays live pane-monitor timer values** (via `getActiveTimers()`) onto the persisted session before returning — reported `activeMs` for session, agents, and cycles may exceed last-flushed persisted values by up to one poll interval; callers cannot distinguish persisted vs live. `companion` protocol request reads/updates companion name and returns full `CompanionState` — the only companion field mutable via protocol (name). **`pane-exited` handler unregisters the pane before calling `handlePaneExited()`** — if the handler throws, the pane is gone from the registry but session state is never updated; subsequent `pane-exited` events for the same pane return `{ok:true}` with no effect. **Disk-recovery pattern**: `resume`, `rollback`, `reopen-window`, and `reconnect` all fall back to loading from disk when the session is not in the in-memory map — each checks `{cwd}/.sisyphus/sessions/{sessionId}/state.json`; `rollback` calls `registerSessionCwd()` (re-persists to registry), while `resume` and `reopen-window` construct a minimal tracking entry without registering. **`reconnect` protocol** re-binds a live tmux session to the daemon after a restart: calls `sessionManager.reconnectSession()` which finds the session by name, re-captures ID and windowId, and updates all in-memory maps — use when daemon restarted mid-session and the tmux session is still running but the registry is stale. **`delete` protocol** kills the session if it's active (kill errors are swallowed — unlike the `kill` handler which propagates), removes the entry from the registry, then removes the entire session directory with `rmSync({ recursive: true, force: true })` — irreversible, no confirmation.
- **session-manager.ts** — Entry point for all session operations. **Lifecycle**: `startSession()` (create + spawn orchestrator, fire-and-forget Haiku naming with pane label updates — retries up to 5 suffixed variants on collision, gives up silently if all taken), `resumeSession()` (reuse or recreate tmux, mark lost agents — **lost-agent detection is conditional on `session.tmuxWindowId` being non-null**: if windowId is missing, e.g., daemon crashed between `registerSessionCwd()` and `registerSessionTmux()`, all agents remain `running` in state but are unreachable with no recovery path short of `handleKill()`), `reopenWindow()` (recreates if killed), `pruneOldSessions()` (keeps 10 recent or 7-day sessions). **`reconnectSession(sessionId, cwd)`** re-binds a daemon-untracked session to a live tmux session by name — unlike `resumeSession()` which may create a new tmux session, `reconnectSession()` throws if the tmux session doesn't exist by name (directs user to `sisyphus resume`); it re-captures `$N` ID and windowId, calls `initSessionMeta()`, then updates `registerSessionTmux()` / `trackSession()` / `updateTrackedWindow()` in order. **`resetAgentCounterFromState()` is NOT called in `reconnectSession()`** — if the orchestrator spawns agents after reconnect, the in-memory counter starts at 0 and generates `agent-001` colliding with pre-reconnect agents already in state; `resumeSession()` is the only path that resets the counter. **Handlers**: `handleSpawn()` (re-activates `completed` sessions before spawning — allows orchestrator to spawn agents after a session is marked completed mid-cycle), `handleSubmit()`/`handleReport()`, `handleYield()` (re-activates `paused` sessions before proceeding, then guards against pane monitor races), `handleComplete()` (flushes timers + **full teardown**: untrack, unregister panes, clear agent counter, delete orchestratorDone, switch to home session, kill tmux session — mirrors `handleKill()`; add any new complete path here too or it leaks), `handleKill()` (atomic cleanup), `handleContinue()` (re-activates a `completed` session — delegates to `state.continueSession()`; the only protocol path that transitions out of `completed` without spawning an agent), `handleRestartAgent()`/`handleKillAgent()`, `handleRollback()` (validates cycle, kills running agents, restores snapshot, cleanup), `handlePaneExited()` (unexpected exits → respawn or notifications). **Respawn coordination**: `onAllAgentsDone()` uses `pendingRespawns` + `orchestratorDone` guards to prevent concurrent respawns; creates cycle snapshot, ensures window exists, respawns orchestrator via `setImmediate`. Inside `setImmediate`: if session is `paused` AND still in `respawningSessions` (pane monitor raced during yield→respawn window), it's re-activated then respawned; if `paused` WITHOUT the guard, the guard is cleared and respawn is skipped (legitimate pause, not a race). **`pendingRespawns` has no TTL** — if the `setImmediate` callback throws after `pendingRespawns.add()` but before `pendingRespawns.delete()` (e.g., snapshot creation failure), the session is permanently blocked from all future respawns; recovery requires `handleKill()` + `startSession()`. **Startup double-trigger**: `recoverSessions()` calls `onAllAgentsDone()` immediately for stuck sessions, and the first pane-monitor poll cycle independently detects them as stuck and calls it again; `pendingRespawns` guard absorbs the second call — no double respawn — but be aware that any logging or metrics around `onAllAgentsDone` entries will fire twice. **`pruneOldSessions()` silently skips directories with unreadable state files** — corrupted session dirs (state.json missing or invalid JSON) are excluded from the deletion candidates and accumulate on disk without warning. **Companion hooks**: fire at `startSession` (`onSessionStart`), `handleSpawn` (`onAgentSpawned` + async `generateNickname` written back via `state.updateAgent` — if the session completes before Haiku responds, the `updateAgent` call silently discards), `handleComplete` (`onSessionComplete` — detects level-up + achievements, then calls **`fireCompletionCommentary(events)`** with all triggered events ordered `[session-complete, level-up?, achievement?]`; **all events with non-null commentary become pages in `showCommentaryPopupQueue()`** — Enter advances through pages; `session-complete`, `level-up`, and `achievement` each show if Haiku returns text; also fires `generateSentiment` fire-and-forget which may call `writeSessionSummary` a second time to add the `sentiment` field after Haiku responds), `handlePaneExited` (`onAgentCrashed`). All hooks are fire-and-forget in try/catch; errors never propagate into the handler. **`fireCommentary()` helper** (used by `session-start`, `cycle-boundary`, `agent-crash`): the `companion` arg passed to `generateCommentary` is prompt context only — the `.then()` callback calls `loadCompanion()` fresh then `recordCommentary()` (writes both `lastCommentary` and `commentaryHistory`), so saves never stomp in-flight state mutations. Text is saved directly with no post-processing clamp — schema enforces ≤160 chars at Haiku call time. `flash=true` triggers `showCommentaryPopup()` for `cycle-boundary`; `session-start` and `agent-crash` fire without a popup. **Completion events do NOT use `fireCommentary()`** — they use `fireCompletionCommentary()` → `showCommentaryPopupQueue()` with all events as pages. **`cycle-boundary` commentary context** includes goal, spawned-agent summary, and — if `cycle-N.md` exists in `logsDir` — the first 200 chars of that log file (read inside a best-effort try/catch; silently absent when missing); this is the only place session-manager reads orchestrator-written cycle logs.
- **orchestrator.ts** — Spawns/respawns orchestrator Claude each cycle with mode-specific system prompts (base + strategy/implementation/validation/completion suffixes; strategy default). Injects available agent types, caches window/pane IDs, writes prompts to `prompts/` subdir. Formats session state: goal, context, messages, cycle history (agent reports + status), repositories (branches, dirty flags, agent assignments). `detectRepos()` scans for git repos and filters by config.repos if present. **`modeContentBuilders` map** (`Record<string, ModeContentBuilder>`) extends the *user* prompt (not system prompt) with mode-specific content — currently only `completion` is registered. `buildCompletionContent()` appends an agent summary table (id/name/type/status/final-report-summary), inline `cycle-*.md` log files from `logsDir`, and a reference path to `reportsDir` — all read at prompt-build time, so missing dirs/files are silently skipped. To add content for a new mode, add a key to `modeContentBuilders`; the base state formatting is shared across all modes.
- **agent.ts** — Spawns agent Claude instances (Anthropic Claude or OpenAI Codex per provider); auto-increments per-session counter. Resolves agent type configs via frontmatter (model, color, effort, interactive). Creates Claude plugins with conditional hooks: `Stop` hook required for non-interactive agents; agent-type-specific `UserPromptSubmit` hooks for (problem, plan, requirements, review, review-plan, debug, operator, test-spec, explore). Supports sub-agent definitions: copies subdirectories alongside agent type files. Handles agent restart and report submission. **`allAgentsDone()` checks ALL running agents in the session** — agents can call `sisyphus spawn` to create sub-agents, and the orchestrator won't respawn until every running agent (including agent-spawned ones) finishes; also requires `session.agents.length > 0` — zero-agent sessions never trigger all-done. **`createAgentPlugin()` substitutes `$SISYPHUS_SESSION_DIR` and `$SISYPHUS_SESSION_ID` via literal string replacement in agent type template bodies** at plugin-creation time (before writing to disk) — templates can use these as placeholders; the substitution happens before the file lands in the plugin dir, so claude never sees the raw variable names. **`resolveRequiredPluginDirs(cwd)`** (from `plugins.js`) returns project- and user-level plugin dirs that are injected as extra `--plugin-dir` flags alongside the agent's own plugin — agents always have access to both their per-agent plugin and any registered project plugins; order of flags is agent plugin first, then required dirs. **`claudeSessionId`**: Anthropic agents get a fresh `randomUUID()` per spawn passed as `--session-id`; a new UUID is generated on restart (doesn't resume the prior Claude conversation). OpenAI agents have no session ID. **OpenAI and Anthropic agents use fundamentally different prompt delivery paths**: OpenAI writes a single merged file `{agentId}-codex-prompt.md` containing `agentConfig.body + suffix + task` joined with double newlines, passed via shell substitution to codex; Anthropic delivers the body via the plugin's `agents/` subdir (written by `createAgentPlugin()`) and passes the combined body+suffix via `--append-system-prompt` by default, or `--system-prompt` when `frontmatter.systemPrompt === 'replace'` — `replace` mode overwrites Claude's entire default system prompt rather than appending; with the task inline as the final argument. **`{{WORKTREE_CONTEXT}}` placeholder** in agent suffix templates is replaced with an empty string in `renderAgentSuffix()` — reserved for future worktree injection; templates using it produce no output for that placeholder today. `resumeArgs` is `undefined` for OpenAI agents (the field is set only in the Anthropic branch of `setupAgentPane()`); `restartAgent()` reconstructs the full command via `setupAgentPane()` in both cases so the undefined value is never used. **`resetAgentCounterFromState(sessionId, agents)`** is called by `resumeSession()` to recover the in-memory counter from existing agent IDs (scans for max `agent-NNN` suffix) — prevents new agents getting duplicate IDs after daemon restart; if no agents match the pattern, silently resets to 0. **`restartAgent()` zombie detection**: before restarting an agent with `status === "running"`, calls `tmux.paneExists()` — if the pane is dead but state says running, marks it `lost` silently and proceeds; `running` status does NOT guarantee pane existence. **Last-agent pane kill is deferred**: `handleAgentSubmit()` kills a completed agent's pane immediately for all except the last — the last agent's pane is NOT killed at submit time, only inside `onAllAgentsDone()`'s `setImmediate` callback; this prevents tmux from destroying the window when the last pane is killed (detaches the user). **`nextReportNumber()` counts only non-`-final.md` files** — interim report numbers 001, 002… are independent of the final report (`agentId-final.md`); the final report never increments the interim counter.
- **haiku.ts** — Shared Haiku SDK wrapper (imported by `summarize.ts`, `companion-commentary.ts`, `companion-popup.ts`). `callHaiku(prompt, systemPrompt?)` returns raw string output or null; the optional `systemPrompt` is forwarded directly to the SDK `query()` options — pass it when you need a terse system directive (e.g., `summarize.ts` uses `'You are a concise text generator...'` to suppress preamble). `callHaikuStructured<T>(prompt, jsonSchema, zodSchema)` takes **both** a raw JSON schema object (passed as `outputFormat` to the API — caller must provide this, there is no internal `zodToJsonSchema` conversion) **and** a Zod schema for `safeParse` validation. Reads `msg.structured_output` from the result message. **`safeParse` failure returns `null` without logging**; a model response that is valid JSON but wrong shape is indistinguishable from model failure or auth cooldown. Module-level `disabledUntil` gates both functions on 401/403 — one auth failure silences ALL Haiku calls for 5 minutes across all callers. **`callHaiku` uses `maxTurns: 1`; `callHaikuStructured` uses `maxTurns: 2`** — the structured call may need a second turn for the model to emit `structured_output` via tool use; callers cannot override this.
- **summarize.ts** — Optional async utilities via Haiku SDK: `generateSessionName()` creates kebab-case session names; task is **sliced to 500 chars** before sending; result is **sliced to 30 chars** (can break mid-hyphen); validated against `/^[a-zA-Z0-9_-]+$/` — multi-word or non-ASCII Haiku output returns `null` silently, causing session-manager to exhaust all 5 collision retries and keep the UUID-based name. `summarizeReport()` slices report to **3000 chars** before sending. `generateSentiment(input: SentimentInput)` generates a one-sentence emotional read on a completed session — takes a structured object (`task`, `completionReport?`, `agentCount`, `cycleCount`, `crashCount`, `activeMs`, `messages`) rather than raw strings; task sliced to **500 chars**, completion report to **500 chars**, first 5 messages each to **300 chars**, stats injected as a formatted line, result sliced to **200 chars**; called fire-and-forget from `handleComplete()` and enriches the already-written session summary via a second `writeSessionSummary()` call (the `sentiment` field is nullable and absent from the first write). Uses a dedicated `SENTIMENT_SYSTEM` prompt (`'You are a concise text generator…'`) to suppress Haiku preamble. All three share the 5-minute cooldown on auth failures (module-level in `haiku.ts`).
- **frontmatter.ts** — Parses YAML frontmatter from agent type definitions. Extracts metadata (name, model, color, skills, permissionMode, effort, interactive, description, systemPrompt). Detects provider (Anthropic vs. OpenAI) from model. Resolves agent type paths via `resolveAgentTypePath()`. **Resolution order differs by namespace**: unnamespaced types search `project (.claude/agents/) → user (~/.claude/agents/) → bundled (pluginDir/agents/)`; namespaced types (e.g. `sisyphus:name`) search `bundled pluginDir → installed plugin only` — project and user dirs are skipped entirely for namespaced lookups. **`findPluginInstallPath()` reads only the flat registry format** (`registry[key].installPath`) while `discoverAgentTypes()` also handles the v2 nested format (`registry.plugins[key]` and array entries) — if the Claude Code plugin registry uses v2 format, `resolveAgentTypePath()` for that namespace silently returns null while `discoverAgentTypes()` still lists it. `parseAgentFrontmatter()` uses line-by-line regex — inline YAML arrays (`skills: [a, b]`) are silently ignored; only indented-list format (`- item`) is parsed.
- **pane-monitor.ts** — Background poller (5s interval, sleep-aware elapsed time calculation). **Pane liveness**: detects unexpected pane kills, stuck sessions (all agents done + no orchestrator), marks running agents as lost; invokes respawn callback via `onAllAgentsDone()`. **Active time tracking**: maintains in-memory timers (`activeTimers` map) for sessions/agents/cycles; `initTimers()` called at session start AND lazily inside `pollSession()` if the entry is missing (guards against races where a session begins polling before `trackSession` triggers `initTimers`); `pollSession()` accumulates milliseconds, `flushTimers()` persists deltas to state. Tracks sessions with `trackSession/untrackSession` lifecycle. **Sleep detection**: if elapsed > 3× poll interval, only the poll interval is credited (not actual elapsed) — masks machine sleep to prevent inflated totals. **`activeTimers` IS flushed on graceful shutdown** (SIGTERM/SIGINT) — `index.ts` iterates `getTrackedSessionIds()` and calls `flushTimers()` before stopping; time is only lost on daemon crash (not clean restart). **Companion mood**: recomputed each poll cycle from `MoodSignals`; **saved only when mood changes** — `computeMood` mutates `debugMood` on the companion object even when the mood string is unchanged, but `saveCompanion` is only called inside the `if (newMood !== companion.mood)` block; `debugMood` may be slightly stale in companion.json between mood transitions. `recentCrashes` counts agents with `status === 'crashed'` and `completedAt` within last 30 minutes from tracked active sessions. `idleStartTime` clock starts when `activeTimers` becomes empty (not when `trackedSessions` empties) — resets when any timer is re-added. Mood recompute is **throttled to once per minute when idle** (no tracked sessions — `trackedSessions.size === 0`), every 5s poll when active. `idleStartTime`/`idleDurationMs` uses `activeTimers.keys().length === 0` — a different idle signal. These can diverge: if a session is untracked but leaves stale `activeTimers` entries, the throttle skips recompute while `idleDurationMs` stays 0. `justCompleted`, `justCrashed`, `justLeveledUp` use **2-minute temporal decay** via module-level timestamps (`lastCompletionTime`/`lastCrashTime`/`lastLevelUpTime`) — session-manager calls `markEventCompletion()`/`markEventCrash()`/`markEventLevelUp()` exports to set them; pane-monitor checks `(nowMs - lastXTime) < 120_000`. `signals.cycleCount` is computed locally inside each `pollAllSessions()` call as `maxCycleCount` across tracked sessions — `updateCycleCount()`/`resetCycleCount()` exports were removed; session-manager no longer needs to call them. **`idle-wake` commentary**: fires when transitioning from idle (>1 min) to active (`timerKeys.length > 0` while `idleStartTime > 0`) — 50% chance; reloads companion fresh before saving. **`late-night` commentary**: fires during 2–6am polls when `trackedSessions.size > 0`; throttled to once per 30min via module-level `lastLateNightCommentary` timestamp; calls `showCommentaryPopup(text)` on success. **Late-night context includes task snippets**: iterates tracked sessions, appends each task truncated to 80 chars; stops appending (not aborts) once context exceeds 300 chars total — sessions beyond the cutoff are silently excluded from the prompt. **`flushTimers()` has two silent early-return paths**: (1) session has no `activeTimers` entry — no-op; (2) session is not in `trackedSessions` — no-op; any time accumulated in `activeTimers` for an untracked session is permanently lost without warning. Per-poll `pollSessionCache` is populated during `pollSession()` and reused for mood signal computation — avoids re-reading state files for sessions already polled in the same cycle. **`completed` session cleanup**: `pollSession()` has a special path for `status === 'completed'` sessions — if the tracked orchestrator pane is gone (or no orchestrator was ever registered), calls `cleanupSessionMaps()` + `untrackSession()` and returns early; this is how completed sessions leave `trackedSessions` between `handleComplete()` teardown and the next poll cycle. Sessions that complete during the poll window are cleaned up within one poll interval, not immediately. **Sessions with `windowId === null` are silently skipped** in the `pollAllSessions()` loop (`if (windowId)` guard) — no pane polling, no timer accumulation, and no mood signal contribution until `updateTrackedWindow()` sets a value; this is the normal state between `trackSession()` and the orchestrator spawning its first pane.
- **pane-registry.ts** — Central registry mapping paneId → {sessionId, role, agentId}. Enables fast pane lookup and exit notification routing.
- **respawn-guard.ts** — Tracks sessions in yield→respawn transition via `respawningSessions` set to prevent pane monitor from pausing the session while orchestrator is being respawned.
- **notify.ts** — Desktop notifications via a **long-lived singleton process** (`SisyphusNotify.app`) that accepts JSON lines on stdin — not a per-notification subprocess. Three-tier fallback: native app → `terminal-notifier` → `osascript`. Native app must be installed separately at `~/.sisyphus/SisyphusNotify.app/` (not bundled). Click-to-switch is **iTerm2-only** (AppleScript); switch script written lazily to `~/.sisyphus/notify-switch.sh`. tmux socket hardcoded to `/tmp/tmux-{uid}/default` — fails with non-default socket paths. Switch script hardcodes `/opt/homebrew/bin/tmux` — silently fails on Intel Macs (Homebrew at `/usr/local/bin`).
- **status-dots.ts** — Computes per-session phase indicators for a tmux dashboard window. **Phase detection derives orchestrator pane from persisted cycle state** (`OrchestratorCycle.paneId` + `!completedAt`) — does NOT use the in-memory `sessionOrchestratorPane` map from `orchestrator.ts`, so it survives daemon restarts without recovery. Claude hook state files at `/tmp/claude-tmux-state/{paneNumericId}` (written by Claude Code hooks, not sisyphus) distinguish `orchestrator:processing` from `orchestrator:idle`. Both `stopped` and `idle` → `orchestrator:idle`; `processing` (or missing file) → `orchestrator:processing`. **`between-cycles` phase has two triggers**: (1) `respawningSessions.has(sessionId)` (from respawn-guard.ts) — the expected yield→respawn window; (2) fallback when orchestrator pane is gone and no agents are running but session is still `active` — this is the "limbo" state after an unexpected orchestrator exit before pane-monitor fires `onAllAgentsDone()`. `recomputeDots()` **groups sessions by cwd** — finds the dashboard window via `@sisyphus_dashboard` option on the home session (30s cache), then writes `@sisyphus_dots` as a tmux format string to that window **as a window option** (`setWindowOption`). **`@sisyphus_phase` is no longer written to the tmux session option** — `recomputeDots()` only populates the in-process `sisyphusPhases` map; external tools (tmux format strings, shell scripts) that read `#{@sisyphus_phase}` will get stale/unset values. `sessionNamesWithPhase` set was added as infrastructure for future cleanup of stale phase options — no cleanup logic exists yet. Exports `DOT_MAP` (phase→icon/color), `getSisyphusPhases()` (populated each cycle), and `getTotalActiveSessionAgents()` for `segments/compositor.ts` — **owned here, not in pane-monitor**; counts **all agents** (any status) in sessions with `status === 'active'` — not just `running`-status agents — reset to 0 at the start of each `recomputeDots()` call (returns 0 before first cycle). `markSessionCompleted()` adds a 5-min TTL entry so completed dots persist after the session leaves active tracking — called on `handleComplete()` only, not `handleKill()`. **If a session is continued/resumed within the 5-min TTL**, it re-enters `trackedSessions` while the TTL entry still exists — `recomputeDots()` renders both, producing a duplicate dot for that cwd; clears itself within one TTL expiry (no explicit invalidation on resume). `setTrackedEntriesProvider()` must be called at startup or `recomputeDots()` is a no-op. **`invalidateDashboardCache(cwd)` is exported but never called** — the 30s dashboard window cache is effectively permanent-until-daemon-restart for any cwd once set; any future code path that reassigns `@sisyphus_dashboard` must call it or dots will write to the stale window. **Completed-session dots require a live home session** with `@sisyphus_dashboard` set at render time — if the user closed the dashboard window after `handleComplete()`, the dot silently disappears even within the 5-min TTL (no error, `getDashboardWindowId` returns null and the write is skipped). **`recomputeDots()` skips tracked entries with `windowId === null`** (`if (!entry.windowId) continue;`) — the session is absent from `byCwd` grouping and gets no dot until a windowId is registered. **Dots within a cwd group are stable-sorted by `session.createdAt`** — order is consistent across daemon restarts because `createdAt` is persisted; dots for the same cwd always appear oldest-first regardless of tracking order.
- **segments/** — Pluggable status bar system replacing the old `status-bar.ts` monolith. `Compositor` class (in `segments/compositor.ts`) owns all rendering. Built-in segments: `sessions` (normal tmux sessions), `sisyphus-sessions`, `companion`, `windows` (per-session window tabs), `session-name` (clock/label). **Left vs right asymmetry**: right side writes one global `status-right` string; left side writes `@sisyphus_left` as a session option on *every* tmux session then sets global `status-left` to `#{E:@sisyphus_left}` — O(n sessions) tmux writes per render cycle, and left content is session-specific (each session sees its own window list). **`initialize()` fires once on first `render()` call** — blanks `window-status-format`/`window-status-current-format`/`window-status-separator` and sets `status-left-length`/`status-right-length` to 250; manually changed tmux settings are overwritten on first render after daemon start. **Config built at startup in `index.ts`**: `statusBar.segments` and `statusBar.colors` do a shallow key-merge with defaults; `statusBar.left`/`statusBar.right` fully replace the default segment order when present. **External segments** (for third-party or protocol-driven content): `registerExternal()` adds; `updateExternal()` updates content — throws if segment not pre-registered (unlike `unregisterExternal()` which silently no-ops). Protocol requests `register-segment`/`update-segment`/`unregister-segment` reach the compositor via a reference stored in `server.ts` via `setCompositor()`. **Segment ordering**: config `left`/`right` arrays determine order; segments not in those arrays fall back to `priority` ascending. **`includesArrows: true`** in `SegmentOutput` lets a segment own its own powerline arrows — compositor skips the closing arrow on the last band and skips the entry arrow for that segment; both `sessions.ts` and `sisyphus-sessions.ts` use this via `Compositor.renderSessionBand()`. **Caches live in `compositor.ts`**: session-order file (`~/.config/tmux/session-order`) cached 30s; companion state cached 10s — `saveCompanion()` calls from session-manager lag up to 10s in the status bar. **`compositor.render()` is called from `dotsCallback` in `index.ts`** alongside `recomputeDots()` and `writeManifest()` — all three fire on each pane-monitor poll cycle; render errors are swallowed (best-effort try/catch). **tmux format comma constraint applies to `status-right`/`status-left` strings only**: `#[fg=x,bold]` combined attributes silently break `#{?}` conditionals — use separate `#[fg=x]#[bold]` blocks; this does NOT apply to `pane-border-format` which uses backslash-comma escaping instead. **Sisyphus session classification** uses `getSisyphusPhases()` from `status-dots.ts` — a session absent from the phases map is silently dropped even if it has an `ssyph_` prefix. **Only normal sessions are ordered** by the session-order file; sisyphus sessions render in `phases.values()` iteration order. **Requires Nerd Fonts** — missing font renders powerline arrows as boxes with no error or fallback.
- **state.ts** — Session creation, mutation, and persistence. Uses **session-level mutex** (`withSessionLock`) to prevent read-modify-write races during concurrent updates. **Lock is per-sessionId only** — operations on different sessions run concurrently; callers that assume global ordering (e.g., reading one session while writing another) are not protected. Atomically writes state via temp file + rename. **Manages cycle snapshots**: `createSnapshot()` (at cycle boundaries), `restoreSnapshot()` (restore to prior state), `listSnapshots()` (available snapshots), `deleteSnapshotsAfter()` (cleanup post-rollback). **`updateAgent()` searches in reverse** — `.slice().reverse().find()` — so if multiple entries share the same `agentId`, the most-recently-appended one is updated; callers that push duplicate IDs will silently update the wrong record if they add rather than mutate. **`restoreSnapshot()` clears `tmuxSessionName` and `tmuxWindowId`** from the restored state — the existing tmux session/window is NOT killed, just orphaned; `resumeSession()` must be called afterward to re-bind the session to tmux.
- **tmux.ts** — tmux CLI wrapper: session/window/pane creation and lifecycle. **All tmux `-t` targets use the `t()` helper** (`shellQuote()` → single-quote wrapping) — this is required because `$34`-style session IDs contain `$` which gets shell-expanded when passed to `/bin/sh` with double-quote wrapping; never revert to `"${target}"` double-quote style on any target string. **`resolveSessionId()` uses `list-sessions` + exact name match** (not `display-message -t "=name"`) — the old `display-message` approach requires an attached client and silently fails in headless daemon context; if you need a session's `$N` ID from its name, use `resolveSessionId()`, do not roll your own. **`getFirstWindowId(sessionTarget)`** returns the index-0 window ID for a session target — used by recovery to discover a windowId when state was partially lost. **Session defaults**: every `createSession()` calls `configureSessionDefaults()` which sets `pane-border-status top`, disables `allow-rename`/`automatic-rename` (prevents tmux from clobbering pane titles we set), and installs `after-kill-pane`/`pane-exited` hooks that auto-retile to `even-horizontal`. Hooks are set at **session scope** — they fire for all windows in that session, not just the initial window. The window is named `"main"` at `new-session -n main` creation time — `configureSessionDefaults()` does NOT rename it afterward. **Session existence API — three functions, each with a distinct use case**: `sessionExistsById(tmuxSessionId)` uses `has-session -t "$N"` — safe because `$N` is always `$<integer>` and tmux matches it exactly (no prefix risk); `sessionNameTaken(name)` uses `list-sessions` exact match — use only for collision detection at creation/rename; `isSessionAlive(id, name)` dispatches to `sessionExistsById` when id is set, else `sessionNameTaken` — the standard check for "is this session still running". Never use `has-session -t` directly with session names — it does prefix matching and returns true for `"foo"` when only `"foo-resume"` exists. **`initSessionMeta(target, cwd, sisyphusSessionId)`** sets `@sisyphus_cwd` and `@sisyphus_session_id` on every newly created session — required for `findHomeSession()` to find the home session and for the TUI session selector. **Pane creation**: `createPane()` targets the first/last existing pane (not window) to guarantee absolute left/right placement — using the window as target splits relative to the active pane. **Home session support**: `findHomeSession()` matches by `@sisyphus_cwd` session option, not by name — makes one `show-options` tmux exec per session (O(n)) on each call; a non-`ssyph_` session that hasn't set this option is invisible to it; `switchAttachedClients()` moves attached clients to home session before destroying sisyphus session. **Option helpers**: `getGlobalOption()`, `setGlobalOption()`, `setSessionOption()`, `unsetSessionOption()` all use `execSafe` — none throw; writes to missing session targets silently no-op. `listAllSessions()`, `listAllPanes()` — used by `segments/compositor.ts` for whole-server status rendering. **Pane styling**: `setPaneStyle()` stores metadata as per-pane user vars (`@pane_role`, `@pane_session`, `@pane_cycle`, `@pane_mode`, `@pane_color`) and `pane-border-format` — all set with `-p` (per-pane flag) so each pane has its own label format; window-level `pane-border-style` uses `#{@pane_color}` which resolves per-pane at render time (last-write-wins for the window option itself). The `modeSegment` in `pane-border-format` uses `#[fg=color\,italics]` (backslash-comma) inside a `#{?}` conditional — this is the correct tmux escape for commas within conditionals in `pane-border-format` context; do not "fix" it to a space-separated block. The `status-right`/`status-left` rule (separate `#[fg=x]#[bold]` blocks) applies to global status format strings, not `pane-border-format`. `updatePaneMeta()` updates individual vars without rebuilding the full format string (used by auto-naming).
- **companion.ts** — XP/level/mood/achievement system persisted to `~/.sisyphus/companion.json` via its own temp+rename (not `state.ts` mutex). `loadCompanion()` applies forward-compat field fills on read — includes `spinnerVerbIndex` (defaults to 0), `consecutiveHighCycleSessions` (defaults to 0), `commentaryHistory` (defaults to []), and `baselines: defaultBaselines()`; this fill runs on every load so adding new fields here is safe across daemon upgrades. **`recordCommentary(companion, text, event)`** is the canonical write path for any commentary — mutates the companion in-place, writes `lastCommentary` AND appends to `commentaryHistory` ring buffer (capped at 30 entries); caller must call `saveCompanion()` afterward. Direct assignment to `companion.lastCommentary` bypasses the ring buffer and breaks `buildHistoryContext()` dedup in `companion-commentary.ts` — never write `lastCommentary` directly. `computeMood()` with no `signals` arg returns hour-based mood only (existential 02–06, sleepy 22–02, zen otherwise) — signals arg required for full scoring. `computeMood()` also reads `companion.recentCompletions` directly for "flow state" scoring (last completion within 30 min + active session) — a cross-session signal baked into the companion object, not passed via `MoodSignals`. **`computeMood()` fully uses z-scores**: calls `zScore()` for all four baseline metrics (`sessionZ`, `cycleZ`, `agentZ`, `dailyZ`) and uses them in every mood's scoring formula — active code, not future infrastructure. `zScore()` uses cold-start defaults when `count < MIN_SAMPLES (5)`, with a 20%-of-mean stddev floor and per-metric absolute floors. **Tie-breaking**: iterates `['happy', 'grinding', 'frustrated', 'zen', 'sleepy', 'excited', 'existential']` with strict `>`; `0 > -1` fires on first iteration — `happy` wins all-zero ties (not `grinding`, despite the initializer). `onSessionComplete()` returns an array of newly unlocked `AchievementId`s (empty if none) — callers must check this to trigger achievement commentary. XP formula: strength×50 + endurance/3.6M×20 + wisdom×40 + patience×8. **`computeStrengthGain(agentCount)`** is the tiered strength credit function (1/2/3/4/5 points for ≤2/≤5/≤10/≤20/>20 agents) — `onSessionComplete()` uses this instead of `strength++`; delta-safe via `session.companionCreditedStrength` (written by session-manager at completion alongside `companionCreditedCycles`/`companionCreditedActiveMs`). Level thresholds: base 150 XP, each level multiplies by 1.35; `getTitle()` scans `TITLE_MAP` backward — levels 21–24, 26–29, 31+ inherit the nearest lower defined title. `computeLevelProgress()` exported for progress bar rendering: returns `{xpIntoLevel, xpForNextLevel}`. **`onSessionComplete()` is delta-safe** against continue→re-complete double-crediting: reads `session.companionCreditedCycles`, `session.companionCreditedActiveMs`, and `session.companionCreditedStrength` (all written by `session-manager.ts` at completion) and credits only the delta; missing fields default to 0 (full credit on first completion). **`computeWisdomGain()` is NOT delta-safe** — there is no `companionCreditedWisdom` field; if `onSessionComplete()` fires twice for the same session (continue→re-complete), wisdom points are double-credited; this is asymmetric with strength/endurance/patience which all have delta guards. **`computeWisdomGain()` awards 0–3 wisdom points per session**: +1 for ≥80% of agents completing without crash/kill/lost; +1 for ≥2 agents per cycle average (parallelization); +1 for ≥2 distinct orchestrator modes used (`mode` field on cycles). No stddev gate; `isEfficientSession()` no longer exists. **Patience uses sqrt formula**: credits `ceil(sqrt(totalCycles)) − ceil(sqrt(creditedCycles))` points (diminishing returns) plus +1 for first new `validation` mode and +1 for first new `completion` mode seen in the session. **Welford baseline accumulation**: each `onSessionComplete()` feeds `activeMs`, `totalCycles`, and `agents.length` into per-metric `RunningStats` (`sessionMs`, `cycleCount`, `agentCount`) via `welfordUpdate()`. `sessionsPerDay` uses a `pendingDayCount` accumulator finalized into Welford on the first completion of a new calendar day; gap days between `lastCountedDay` and yesterday are zero-fed. **Sessions on a day that sees no subsequent completion day are permanently stuck in `pendingDayCount` and never enter the `sessionsPerDay` distribution.** **`consecutiveEfficientSessions`** increments when `totalCycles <= 3`; resets on `totalCycles > 3` — unlocks `speed-demon` achievement at ≥10 (renamed from `iron-will`). **`consecutiveHighCycleSessions`** increments when `totalCycles >= 8`; resets otherwise — unlocks new `iron-will` achievement at ≥5. Both are separate from `wisdom` stat and crash tracking. **`onAgentCrashed()`** only resets `consecutiveCleanSessions`; `sessionsCrashed` is incremented once per session inside `onSessionComplete()` based on whether any agent has `status === 'crashed'`, not per individual crash event. `recentCompletions` capped at 10 entries; used by `momentum` (5 completions within 4h) and `overdrive` (6+ in one calendar day from recentCompletions). **`normalizeTask()`** keys `taskHistory` as `{cwdBasename}:{task[:100].toLowerCase().trimCollapsed}` — tasks differing only in case or whitespace normalize to the same key (intentional); repos sharing a basename collide across projects. **`updateRepoMemory()` never updates `moodAvg`** — initialized to 0 on first visit and never changed on any subsequent event; `generateRepoNickname` always receives `moodAvg: 0`, which classifies every repo as "mostly negative" (threshold > 0.3 is never met).
- **companion-commentary.ts** — Haiku-generated 1–2 sentence, ≤160-char commentaries for lifecycle events via `callHaikuStructured()` with a hardcoded `COMMENTARY_JSON_SCHEMA` object (for the API's `outputFormat`) plus `CommentaryZodSchema` for `safeParse` — prompt instructs 160-char max; **Zod schema enforces `.min(10).max(160)` — responses under 10 chars OR over 160 chars cause `safeParse` to fail and `callHaikuStructured()` returns null silently; no clamp fallback** (`clampCommentary()` no longer exists). 5-minute cooldown on 401/403 (module-level `disabledUntil` in `haiku.ts`). `shouldGenerateCommentary()` probabilities: `session-start`/`session-complete`/`level-up`/`achievement`/`late-night`/`cycle-boundary` → always; `idle-wake` → 50%; `agent-crash` → 30%. All non-zero — no event type returns null unconditionally. Time-of-day personality: chipper 06–10, professional 10–17, reflective 17–22, dry-humor 22–02, delirious 02–06. **Raw stat numbers are never injected into Haiku prompts** — `buildPersonality()` converts each stat via tier-bucketed prose descriptions (`STRENGTH_TIERS`, `ENDURANCE_TIERS`, `WISDOM_TIERS`, `PATIENCE_TIERS`; each has 10 thresholds); Haiku sees only the matching prose paragraph, not the numeric value. **`buildSentimentContext()`** calls `getRecentSentiments(3)` from `history.ts` and injects the last 3 session sentiments (task + sentiment string) into the commentary prompt — the only cross-module read from `history.ts` here; produces empty string when history is absent (no error). **`generateNickname()`** takes the first whitespace-split token from Haiku's response and truncates to 20 chars — multi-word responses silently lose everything after the first word. **`nicknameStyleGuide()`**: level ≥ 15 unconditionally returns legendary names regardless of mood/stats, overriding all mood+stat branches. **`generateRepoNickname(repoPath, memory)`** takes a `RepoMemory` struct (from `history.ts`: visits, completions, crashes, moodAvg) and generates a 1–2 word repo nickname; truncated to 30 chars; unlike `generateNickname`, multi-word output is expected — callers must supply the memory object, this function never loads it. **`buildHistoryContext()` uses `slice(-10)` of `commentaryHistory`** (ring buffer capped at 30) — only the last 10 commentaries are shown to Haiku for dedup; entries 1–20 are invisible to the model and can repeat without detection. **Anti-repetition injection per call**: `pickVoiceConstraint()` randomly selects 1 of 16 structural constraints (e.g., "Start with a verb", "Use a question", "Reference a physical sensation") injected into the prompt as a hard constraint for that call; `pickSeedThought()` randomly selects 1 of 15 ambient prompts (e.g., "The terminal cursor is blinking") as optional riff material; `sampleExamples(4)` randomly samples 4 from a pool of 20 examples (shuffled each call) — each call sees a different example subset. These three mechanisms are stateless and re-randomized on every `generateCommentary()` call; there is no way to reproduce the exact same prompt twice.
- **sessions-manifest.ts** — Writes a manifest of all active sessions (sisyphus-managed `S` and home `H`) to disk on each poll cycle via `writeManifest()`, called from `dotsCallback` in `index.ts`. Produces two formats atomically (temp+rename): JSON at `sessionsManifestPath()` and TSV at `sessionsManifestTsvPath()`. TSV has a `#ts:epoch` header line. **Home sessions classified by `@sisyphus_cwd` option** (not name prefix) — queried by session name, not `$N` ID (to avoid `$` shell expansion in `execSync`). Fields: `type`, `tmuxName`, `cwd`, `phase` (from `getSisyphusPhases()` — null for home sessions), `dashboardWindowId`. `writeEmptyManifest()` writes a zero-session manifest on daemon shutdown so consumers see an empty state rather than a stale file. The manifest is the canonical way for external tools (TUI, shell scripts) to discover sessions without hitting the daemon socket.
- **colors.ts** — Color cycling for panes: orchestrator always yellow, agents rotate deterministically.
- **companion-popup.ts** — Renders commentary as a tmux `display-popup` targeting all connected clients. **Daemon runs outside tmux (launchd), so must enumerate attached clients via `tmux list-clients`** — no `TMUX` env var available at daemon runtime. Popup appears top-right of each client (x = clientWidth − 38, y = 0). `INNER_WIDTH = POPUP_WIDTH(38) − 6 = 32` — accounts for 2 border (1 per side) + 2 padding per side (both sides × 3 = 6); this was previously −4, causing text overflow. **Popup height is calculated from the actual newline count in the written content string** (not `lines.length + 4`) — the content string includes a leading blank line, an `\n` between text and hint, and a trailing `\n`, so `content.split('\n').length − 1` gives the exact rendered line count; `−1` removes the split artifact of the trailing newline. **`maxContentHeight` is computed as the max across ALL pages and applied uniformly** — all pages render at the same popup height regardless of their individual content length; a single long page in a multi-page queue forces every other page to that height. Content written to `/tmp/sisyphus-popup-{i}.txt` per page (never deleted — accumulate across daemon lifetimes); shell script at `/tmp/sisyphus-popup.sh` drives a **15s auto-dismiss loop** (`POPUP_DURATION = 15`) — any non-Enter keypress flashes the popup border color, Enter dismisses early. Popup is launched with `-E` (auto-exits when the script finishes) and `-b rounded` border style; both the border (`-S`) and the embedded status bar (`-s`) are colored to `moodColor`. `companionPopup: false` in config skips entirely; config is read at popup time via `loadConfig(process.cwd())`, not cached at daemon start. Face intensity derived from `companion.debugMood?.scores[companion.mood]` — may be 0 if `computeMood()` hasn't run with signals since last daemon start. **All pages in a multi-page queue share the same popup height** (`maxContentHeight` = tallest page across all pages computed before any popup is shown) — a short follow-up page (e.g., a one-line achievement) renders inside a taller-than-needed popup; minimum height is 5 lines. **`showCommentaryPopupQueue()` is synchronous and blocks the caller** — each `execSafe('tmux display-popup ...')` waits for the shell script to exit (up to 15s per client); with N attached clients, popups run serially for up to N×15s total; pane-monitor poll continuations and any code awaiting the same promise chain are deferred for this window.

## Key Patterns

**Respawn guard race prevention**: `respawningSessions` tracks sessions in yield→respawn transition. Pane monitor skips empty-window detection for guarded sessions (window may be temporarily empty between killing orchestrator and spawning new one). Guard added on yield/respawn entry, cleared on completion or error — but **not cleared if `onAllAgentsDone()` returns early** (e.g., session not `active` at entry). An early-return without clearing leaves the session stuck in `between-cycles` forever.

**Active time accumulation**: Pane monitor maintains in-memory timers for each session (session, agents, cycles). On each poll, increments timers by elapsed time if panes are alive (using sleep-aware calculation). `flushTimers()` persists deltas to state file. Called before session completion/kill/rollback.

**Session completion & client switching**: When completing or killing a session via `handleComplete()`/`handleKill()`, `switchToHomeSession()` moves any attached tmux clients to a home session (if available in same cwd) before destroying the sisyphus session. Prevents users from being left in a killed session.

**Session creation**: Creates tmux session internally with deterministic naming (`sisyphus-{name}` if provided, else `sisyphus-{sessionId[:8]}`). Validates custom names against alphanumeric/hyphen/underscore pattern. Auto-generates human-readable session names from task via Haiku (fire-and-forget, failures are silent). Kills initial pane spawned by `tmux new-session` after orchestrator spawns its own.

**Agent spawning**: Increments counter, resolves agent type frontmatter (model, color, effort, interactive), detects provider (Anthropic Claude or OpenAI Codex), creates Claude plugin with conditional hooks per agent type, spawns pane.

**Session resumption**: Reuses existing tmux session if available, else creates fresh one. Uses `listPanes()` to detect live agent panes and mark lost agents. Reuses windowId from state if tmux session still exists. Kills initial pane after orchestrator spawns.

**Window reopening**: `reopenWindow()` recreates tmux window/session if killed (e.g., after orchestrator yields with no running agents), reuses existing IDs if available.

**Respawn on all-agents-done**: When all agents finish, `onAllAgentsDone()` adds session to `pendingRespawns`, creates snapshot of current cycle state, ensures tmux session/window exist (recreates if necessary), then respawns orchestrator via `setImmediate`. Uses `pendingRespawns` + `orchestratorDone` guards to prevent concurrent respawns. `orchestratorDone` is checked both in-memory AND by reading `lastCycle.completedAt` from persisted state — the state fallback handles daemon restart between orchestrator yield and agents finishing. `respawningSessions` guard (via respawn-guard.ts) prevents pane monitor from pausing during yield→respawn window.

**Session recovery on startup**: Registry loads; `active`/`paused` sessions recover. Detects stuck sessions (all agents done, orchestrator pane gone) and immediately triggers respawn.

**Cycle snapshots**: At cycle boundaries, `state.createSnapshot()` captures full session state. `handleRollback()` restores to prior snapshot, killing running agents, removing post-rollback snapshots. `handleRollback()` does **not** restart the session — leaves it in restored state; caller must issue `resumeSession()` explicitly.

**Orchestrator crash branching**: `handlePaneExited(role='orchestrator')` forks on agent count: if agents are running → set guard + wait; if all agents already done → trigger `onAllAgentsDone()`; if `agents.length === 0` → pause session (no respawn target). Completed agent panes are killed **lazily** inside `onAllAgentsDone`'s `setImmediate` callback after the new orchestrator spawns, not at submit time.

**Kill-agent ordering**: `handleKillAgent()` unregisters the pane from `pane-registry` before killing the tmux pane. This prevents pane monitor from seeing an unexpected exit and triggering an unintended respawn for a user-initiated kill.

**Companion write races**: `companion.ts` uses its own load/save (not `state.ts` mutex), so concurrent writes are possible. Async commentary callbacks always call `loadCompanion()` fresh immediately before writing `lastCommentary` to minimize the stomp window — never capture a companion reference across an `await`. Multiple commentary chains (session-complete + level-up + achievement) can overlap; the last writer wins on `lastCommentary`, which is acceptable since each chain writes a different event type.

**Prompts**: System + user prompts written to `prompts/` subdir. Passed via Claude flags to avoid tmux quoting issues.

## Constraints

- Never write state directly — use `state.ts` atomic writes
- Always unregister panes from pane-registry and pane-monitor when killing/completing
- Prompt files must be on disk in `prompts/` subdir
- tmux errors are fatal — propagate, don't swallow
- Session recovery must detect and recover stuck sessions immediately on startup
- Agent type frontmatter must be valid YAML; gracefully fall back if missing or malformed
- Snapshots created at cycle boundaries; rollback validates snapshot existence
- Kill initial pane after orchestrator spawns (tmux creates it but orchestrator needs its own)
- Custom session names must be alphanumeric, hyphens, underscores only
- Session names containing tmux format special characters (`#`, `{`, `}`) break `segments/` compositor format strings silently — no escaping is applied
- Interactive agents (interactive: true) skip the Stop hook requirement; all others require submit hook
- Haiku-based naming/summarization is optional (fire-and-forget); auth failures silence feature temporarily with 5-minute cooldown
- Active time must be flushed to state before session completion/kill/rollback, and is flushed automatically on graceful shutdown — only daemon crashes lose un-flushed time
- Respawn guard must be cleared on respawn completion or error; **`onAllAgentsDone()` has two early-return paths with different guard behavior**: (1) if `orchestratorDone` is not set yet (waiting for orchestrator to yield), returns without clearing — guard stays, session correctly shows `between-cycles`; (2) if `session.status !== 'active'`, deletes the guard before returning — but no respawn fires, leaving the session unpaused from `between-cycles` but stuck with no orchestrator. Path (2) is the dangerous one: recover by kill + restart.
- `orchestratorDone` IS cleared by `resumeSession()`, but that doesn't rescue the stuck state from path (2) above — `onAllAgentsDone()` already returned without respawning, and `resumeSession()` doesn't re-trigger the check. The `lastCycle.completedAt` fallback catches both clean yields AND orchestrator crashes — pane-monitor calls `state.completeOrchestratorCycle()` (writing `completedAt`) when the orchestrator pane disappears mid-session, so after a daemon restart the fallback finds it and respawn proceeds normally. Path (2) recovery is still kill + restart.
- Call `recomputeDots()` (best-effort try/catch) after any handler that changes session phase — omitting it leaves the dashboard stale until the next pane-monitor poll
- Companion writes use own `loadCompanion()`/`saveCompanion()` — never route through `state.ts`. Async commentary callbacks must reload fresh before saving (never close over a captured companion reference across an await)
- `onSessionComplete()` return value (new achievement IDs) must be inspected — achievement commentary only fires if the array is non-empty; the companion is already mutated when the function returns
- `trackSession()` (pane-monitor) must be called before any `updateTrackedWindow()` for that session — throws synchronously if session isn't tracked yet; session-manager calls it at session start and resume, but any server.ts recovery path that calls `updateTrackedWindow()` directly bypasses this ordering
- **`commentaryHistory` ring buffer** (max 30, defaults to `[]` on `loadCompanion()`): `recordCommentary()` in `companion.ts` maintains both `lastCommentary` and this buffer simultaneously. `buildHistoryContext()` in `companion-commentary.ts` sends the last 10 entries as negative examples in every Haiku prompt to prevent repetition. Callers that `saveCompanion()` directly without calling `recordCommentary()` bypass the ring buffer — `lastCommentary` and `commentaryHistory` diverge silently.
- **`lastLateNightCommentary`** (pane-monitor module-level) resets to 0 on daemon restart — the 30-minute cooldown is memory-only; first poll after 2am post-restart fires late-night commentary unconditionally regardless of how recently it fired before the restart.
- **`comeback-kid` achievement** checks `session.parentSessionId != null` but nothing in `session-manager.ts` or `state.ts` currently writes this field — achievement is unearnable through normal use until the field is populated. `SessionState.parentSessionId` is declared optional in `shared/types.ts`.
- **`early-bird` vs `night-owl` asymmetry**: both check `session.createdAt` hour. `night-owl` additionally requires `status === 'completed'`; `early-bird` does not — `early-bird` fires at session creation before 6am regardless of whether the session ever completes or crashes.
- **`pickVoiceConstraint()`** (companion-commentary.ts) randomly selects from 16 constraints with no rotation state — identical constraints can appear in consecutive calls. Expected behavior, but do not introduce rotation state expecting it to persist across daemon restarts.
