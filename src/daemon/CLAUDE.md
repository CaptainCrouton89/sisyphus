# src/daemon/

Unix socket server layer managing session lifecycle, tmux panes, and Claude orchestrator/agent spawning.

## Module Responsibilities

- **index.ts** — Daemon entry point: acquires PID lock, starts server + monitor, recovers active/paused sessions on startup with **stuck-session detection**, handles graceful shutdown (SIGTERM/SIGINT), exposes stop/restart CLI commands. **`restart` branches on launchd**: if `launchctl list com.sisyphus.daemon` succeeds, stops the process and polls up to 3s for launchd to respawn (then exits) — does NOT start in-process; falls back to in-process start only when no launchd service found. `acquirePidLock()` exits with code 0 (not 1) when daemon is already running.
- **server.ts** — Listens on `~/.sisyphus/daemon.sock`, parses JSON line-delimited requests, routes to SessionManager. Maintains in-memory session tracking maps (cwd, tmux, window), persists registry to disk for recovery. **`persistSessionRegistry()` only writes `cwd`** — tmux/window metadata is in-memory only. If the daemon crashes after `registerSessionCwd()` but before `registerSessionTmux()`, the recovered session has cwd but no tmux binding. **`messageCounter` is in-memory, initialized to 0 for all sessions (including resumed ones)** — on daemon restart, IDs restart from `msg-001`, potentially colliding with IDs already written to disk. **`status` request overlays live pane-monitor timer values** (via `getActiveTimers()`) onto the persisted session before returning — reported `activeMs` for session, agents, and cycles may exceed last-flushed persisted values by up to one poll interval; callers cannot distinguish persisted vs live. `companion` protocol request reads/updates companion name and returns full `CompanionState` — the only companion field mutable via protocol (name). **`pane-exited` handler unregisters the pane before calling `handlePaneExited()`** — if the handler throws, the pane is gone from the registry but session state is never updated; subsequent `pane-exited` events for the same pane return `{ok:true}` with no effect.
- **session-manager.ts** — Entry point for all session operations. **Lifecycle**: `startSession()` (create + spawn orchestrator, fire-and-forget Haiku naming with pane label updates — retries up to 5 suffixed variants on collision, gives up silently if all taken), `resumeSession()` (reuse or recreate tmux, mark lost agents), `reopenWindow()` (recreates if killed), `pruneOldSessions()` (keeps 10 recent or 7-day sessions). **Handlers**: `handleSpawn()` (re-activates `completed` sessions before spawning — allows orchestrator to spawn agents after a session is marked completed mid-cycle), `handleSubmit()`/`handleReport()`, `handleYield()` (re-activates `paused` sessions before proceeding, then guards against pane monitor races), `handleComplete()` (flushes timers + **full teardown**: untrack, unregister panes, clear agent counter, delete orchestratorDone, switch to home session, kill tmux session — mirrors `handleKill()`; add any new complete path here too or it leaks), `handleKill()` (atomic cleanup), `handleContinue()` (re-activates a `completed` session — delegates to `state.continueSession()`; the only protocol path that transitions out of `completed` without spawning an agent), `handleRestartAgent()`/`handleKillAgent()`, `handleRollback()` (validates cycle, kills running agents, restores snapshot, cleanup), `handlePaneExited()` (unexpected exits → respawn or notifications). **Respawn coordination**: `onAllAgentsDone()` uses `pendingRespawns` + `orchestratorDone` guards to prevent concurrent respawns; creates cycle snapshot, ensures window exists, respawns orchestrator via `setImmediate`. Inside `setImmediate`: if session is `paused` AND still in `respawningSessions` (pane monitor raced during yield→respawn window), it's re-activated then respawned; if `paused` WITHOUT the guard, the guard is cleared and respawn is skipped (legitimate pause, not a race). **Companion hooks**: fire at `startSession` (`onSessionStart`), `handleSpawn` (`onAgentSpawned` + async `generateNickname` written back via `state.updateAgent` — if the session completes before Haiku responds, the `updateAgent` call silently discards), `handleComplete` (`onSessionComplete` — detects level-up + achievements, each triggers an independent async `generateCommentary` + `flashCompanion` chain), `handlePaneExited` (`onAgentCrashed`). All hooks are fire-and-forget in try/catch; errors never propagate into the handler. **`fireCommentary()` helper**: the `companion` arg passed to `generateCommentary` is prompt context only — the `.then()` callback calls `loadCompanion()` fresh before saving `lastCommentary`, so saves never stomp in-flight state mutations. `flash=true` is set only for `session-complete`, `level-up`, and `achievement`; `session-start` and `agent-crash` generate no flash (and `agent-crash` returns null from `shouldGenerateCommentary()` entirely).
- **orchestrator.ts** — Spawns/respawns orchestrator Claude each cycle with mode-specific system prompts (base + strategy/implementation/validation/completion suffixes; strategy default). Injects available agent types, caches window/pane IDs, writes prompts to `prompts/` subdir. Formats session state: goal, context, messages, cycle history (agent reports + status), repositories (branches, dirty flags, agent assignments). `detectRepos()` scans for git repos and filters by config.repos if present.
- **agent.ts** — Spawns agent Claude instances (Anthropic Claude or OpenAI Codex per provider); auto-increments per-session counter. Resolves agent type configs via frontmatter (model, color, effort, interactive). Creates Claude plugins with conditional hooks: `Stop` hook required for non-interactive agents; agent-type-specific `UserPromptSubmit` hooks for (plan, review, review-plan, debug, operator, test-spec, explore). Supports sub-agent definitions: copies subdirectories alongside agent type files. Handles agent restart and report submission.
- **summarize.ts** — Optional async utilities via Haiku SDK: `generateSessionName()` creates kebab-case session names; task is **sliced to 500 chars** before sending; result is **sliced to 30 chars** (can break mid-hyphen); validated against `/^[a-zA-Z0-9_-]+$/` — multi-word or non-ASCII Haiku output returns `null` silently, causing session-manager to exhaust all 5 collision retries and keep the UUID-based name. `summarizeReport()` slices report to **3000 chars** before sending. Both fire-and-forget with 5-minute cooldown on auth failures (module-level in `haiku.ts`, shared across both functions).
- **frontmatter.ts** — Parses YAML frontmatter from agent type definitions. Extracts metadata (name, model, color, skills, permissionMode, effort, interactive, description). Detects provider (Anthropic vs. OpenAI) from model. Resolves agent type paths (project → user → bundled → installed plugins) via `resolveAgentTypePath()`. Discovers available agent types via `discoverAgentTypes()`.
- **pane-monitor.ts** — Background poller (5s interval, sleep-aware elapsed time calculation). **Pane liveness**: detects unexpected pane kills, stuck sessions (all agents done + no orchestrator), marks running agents as lost; invokes respawn callback via `onAllAgentsDone()`. **Active time tracking**: maintains in-memory timers (`activeTimers` map) for sessions/agents/cycles; `initTimers()` called at session start AND lazily inside `pollSession()` if the entry is missing (guards against races where a session begins polling before `trackSession` triggers `initTimers`); `pollSession()` accumulates milliseconds, `flushTimers()` persists deltas to state. Tracks sessions with `trackSession/untrackSession` lifecycle. **Sleep detection**: if elapsed > 3× poll interval, only the poll interval is credited (not actual elapsed) — masks machine sleep to prevent inflated totals. **`activeTimers` is NOT flushed on daemon shutdown** — only on complete/kill/rollback; time accumulated since the last flush is lost on unexpected daemon exit. **Companion mood**: recomputed each poll cycle from `MoodSignals`; **saved only when mood changes** — `computeMood` mutates `debugMood` on the companion object even when the mood string is unchanged, but `saveCompanion` is only called inside the `if (newMood !== companion.mood)` block; `debugMood` may be slightly stale in companion.json between mood transitions. `recentCrashes` counts agents with `status === 'crashed'` and `completedAt` within last 30 minutes from tracked active sessions. `idleStartTime` clock starts when `activeTimers` becomes empty (not when `trackedSessions` empties) — resets when any timer is re-added. Mood recompute is **throttled to once per minute when idle** (no tracked sessions — `trackedSessions.size === 0`), every 5s poll when active. `idleStartTime`/`idleDurationMs` uses `activeTimers.keys().length === 0` — a different idle signal. These can diverge: if a session is untracked but leaves stale `activeTimers` entries, the throttle skips recompute while `idleDurationMs` stays 0. `justCompleted`, `justCrashed`, `justLeveledUp` use **2-minute temporal decay** via module-level timestamps (`lastCompletionTime`/`lastCrashTime`/`lastLevelUpTime`) — session-manager calls `markEventCompletion()`/`markEventCrash()`/`markEventLevelUp()` exports to set them; pane-monitor checks `(nowMs - lastXTime) < 120_000`. `currentMaxCycleCount` is tracked via `updateCycleCount()`/`resetCycleCount()` exports and flows into `signals.cycleCount`. **`idle-wake` commentary**: fires when transitioning from idle (>1 min) to active (`timerKeys.length > 0` while `idleStartTime > 0`) — 50% chance; reloads companion fresh before saving. **`late-night` commentary**: fires during 2–6am polls when `trackedSessions.size > 0`; throttled to once per 30min via module-level `lastLateNightCommentary` timestamp; calls `flashCompanion(text)` on success. **`flushTimers()` has two silent early-return paths**: (1) session has no `activeTimers` entry — no-op; (2) session is not in `trackedSessions` — no-op; any time accumulated in `activeTimers` for an untracked session is permanently lost without warning. Per-poll `pollSessionCache` is populated during `pollSession()` and reused for mood signal computation — avoids re-reading state files for sessions already polled in the same cycle. **`completed` session cleanup**: `pollSession()` has a special path for `status === 'completed'` sessions — if the tracked orchestrator pane is gone (or no orchestrator was ever registered), calls `cleanupSessionMaps()` + `untrackSession()` and returns early; this is how completed sessions leave `trackedSessions` between `handleComplete()` teardown and the next poll cycle. Sessions that complete during the poll window are cleaned up within one poll interval, not immediately.
- **pane-registry.ts** — Central registry mapping paneId → {sessionId, role, agentId}. Enables fast pane lookup and exit notification routing.
- **respawn-guard.ts** — Tracks sessions in yield→respawn transition via `respawningSessions` set to prevent pane monitor from pausing the session while orchestrator is being respawned.
- **notify.ts** — Desktop notifications via a **long-lived singleton process** (`SisyphusNotify.app`) that accepts JSON lines on stdin — not a per-notification subprocess. Three-tier fallback: native app → `terminal-notifier` → `osascript`. Native app must be installed separately at `~/.sisyphus/SisyphusNotify.app/` (not bundled). Click-to-switch is **iTerm2-only** (AppleScript); switch script written lazily to `~/.sisyphus/notify-switch.sh`. tmux socket hardcoded to `/tmp/tmux-{uid}/default` — fails with non-default socket paths. Switch script hardcodes `/opt/homebrew/bin/tmux` — silently fails on Intel Macs (Homebrew at `/usr/local/bin`).
- **status-dots.ts** — Computes per-session phase indicators for a tmux dashboard window. Phase detection reads Claude hook state files from `/tmp/claude-tmux-state/{paneNumericId}` (written by Claude Code hooks, not sisyphus). Both `stopped` and `idle` map to `orchestrator:idle` (needs input) — only `processing` (or missing file) maps to `orchestrator:processing`. **`between-cycles` phase has two triggers**: (1) `respawningSessions.has(sessionId)` (from respawn-guard.ts) — the expected yield→respawn window; (2) fallback when orchestrator pane is gone and no agents are running but session is still `active` — this is the "limbo" state after an unexpected orchestrator exit before pane-monitor fires `onAllAgentsDone()`. `recomputeDots()` **groups sessions by cwd** — finds the dashboard window via `@sisyphus_dashboard` option on the home session (30s cache), then writes `@sisyphus_dots` (a raw tmux format string `#[fg=color]icon`) to that window; also writes `@sisyphus_phase` per session. Exports `DOT_MAP` (phase→icon/color) and `getSisyphusPhases()` (populated each cycle) for `status-bar.ts`. `markSessionCompleted()` adds a 5-min TTL entry so completed dots persist after the session leaves active tracking — called on `handleComplete()` only, not `handleKill()`. `setTrackedEntriesProvider()` must be called at startup or `recomputeDots()` is a no-op. **`invalidateDashboardCache(cwd)` is exported but never called** — the 30s dashboard window cache is effectively permanent-until-daemon-restart for any cwd once set; any future code path that reassigns `@sisyphus_dashboard` must call it or dots will write to the stale window. **Completed-session dots require a live home session** with `@sisyphus_dashboard` set at render time — if the user closed the dashboard window after `handleComplete()`, the dot silently disappears even within the 5-min TTL (no error, `getDashboardWindowId` returns null and the write is skipped).
- **status-bar.ts** — Renders the complete tmux status bar string and writes it to global option `@sisyphus_status`. Called after `recomputeDots()` on each poll cycle and once on startup. Pipeline: `listAllPanes()` (one tmux exec) → read Claude state files per pane → compute highest-priority state per session (processing > stopped > idle) → separate normal vs sisyphus sessions → order by `~/.config/tmux/session-order` (30s cache) → render tmux format strings with per-client `#{session_name}` conditionals for current-session highlighting → `setGlobalOption`. 3 tmux execs total (list-panes -a, list-sessions, set-option -g). **Early-return guard**: if `listAllSessions()` returns empty, `setGlobalOption` is never called and `@sisyphus_status` remains at its previous value (bar stays stale, not blank). **Powerline section bands**: renders three background bands separated by Nerd Font left-pointing arrows (`\uE0B2` U+E0B2) in fixed order: normal sessions (SESSIONS_BG `#252629`) → sisyphus sessions (SISYPHUS_BG `#36383e`) → companion (COMPANION_BG `#4a4d55`). Bands are conditional — sections with no entries are omitted entirely rather than rendered empty. **Active session highlight**: the focused session uses ACTIVE_SESSION_BG (`#3d3225`, warm amber) with ACTIVE_TEXT; inactive sessions use INACTIVE_TEXT. Both normal and sisyphus bands apply this conditional via `#{?#{==:#{session_name},...}}`. **Separator arrows are also session-conditional** (`renderSessionArrow` takes the current session name + left-neighbor name) — inactive intra-band arrows use `fg=sectionBg` (invisible; sessions run together visually); the arrow *preceding* the active session uses `fg=ACTIVE_SESSION_BG` (amber "nose"); the arrow *following* the active session uses `bg=ACTIVE_SESSION_BG` (amber "tail" — closes the highlight). For cross-band transitions, `renderSectionBoundary` keys on the **last session of the previous band** (`trailingName`), not the first session of the new band — if that trailing session is active, the boundary arrow renders amber. **Requires Nerd Fonts** — missing font renders the arrow as a box/question mark without any error or fallback. Normal sessions show Claude activity dot; **sisyphus sessions always render with label `S`** (not the tmux session name) — if multiple sisyphus sessions are active, they are indistinguishable by label; only the phase icon and active highlight differentiate them. **Sisyphus session classification uses `getSisyphusPhases()` map as the primary signal**, not the `ssyph_` prefix — a session in the phases map renders in the sisyphus band; a session with `ssyph_` prefix that is absent from the phases map is silently dropped from both bands (never rendered). **Only normal sessions are ordered** by the session-order file; sisyphus sessions render in `phases.values()` iteration order. **Companion rendering**: passes `maxWidth: 20` and `agentCount: getTotalRunningAgents()` to `renderCompanion()` from `src/shared/companion-render.ts` with fields `['face', 'boulder']` only — no text in normal mode; companion face is agent-count-aware; text beyond 20 chars is truncated before the tmux format string is built. **tmux format comma constraint**: `#[fg=x,bold]` combined attributes silently break `#{?}` conditional expressions — all style attributes use separate `#[fg=x]#[bold]` blocks; mixing them corrupts the conditional without error. **Companion flash**: `flashCompanion(text, durationMs=10000)` sets module-level `flashUntil` epoch and `flashText`. Each `writeStatusBar()` call checks `Date.now() < flashUntil`; during flash, **the entire status bar is replaced** — all session bands are dropped and only a single companion band (face + flash commentary text) renders; the flash render calls `getCachedCompanion()` (10s TTL) so face/mood during flash may lag behind companion state changes. Flash state is cleared lazily — only on the first `writeStatusBar()` call after expiry (never reset proactively). `flashCompanion` is called from async commentary callbacks, so the flash triggers after Haiku completes, not at the event boundary. **Double-call hazard**: calling `flashCompanion` twice within a flash window overwrites both `flashText` AND `flashUntil` — the second call sets `flashUntil = Date.now() + durationMs`, extending the total window by the gap between calls (e.g., second call 1s into a 10s flash produces an 11s total window, not 10s). **Companion data cache**: `loadCompanion()` is cached for 10s (`cachedCompanion` / `companionCacheTime`) — `saveCompanion()` calls from session-manager won't be reflected in the status bar until the TTL expires; flash vars (`flashUntil`, `flashText`) are module-level and update immediately regardless of cache. **Non-atomic render**: the companion section has its own try/catch and won't blank the bar on failure; earlier pipeline stages (`listAllPanes()`, session rendering) have no try/catch — an exception there skips `setGlobalOption` entirely, leaving `@sisyphus_status` unset (blank status bar). **`ensureStatusRightIntegration()`**: called on the first `writeStatusBar()` invocation; injects `#{E:@sisyphus_status}` into `status-right` immediately before the first `#[fg=` block (before the clock section); if no `#[fg=` found, prepends. Also bumps `status-right-length` to 250 if below that. Includes a migration step: if `status-left` still contains `@sisyphus_status` (old injection location), strips it. If `status-right` already contains `@sisyphus_status`, integration is skipped (treated as already done). **Also sets three window-tab globals on first call**: `window-status-separator ''`, `window-status-format`, and `window-status-current-format` — Powerline window-tab styling using right-pointing arrows (`\uE0B0`, distinct from section dividers `\uE0B2`). The tab format embeds `#(~/.tmux/claude-status.sh '#{window_id}')` (external script, not bundled with sisyphus) and `#{@sisyphus_dots}` — missing script returns empty string silently via tmux `#()`, no error or fallback. `statusIntegrated` is a module-level flag never reset at runtime — if `status-right` is manually changed in tmux after daemon start, neither the status-right injection nor the window-tab format writes will re-fire until daemon restart.
- **state.ts** — Session creation, mutation, and persistence. Uses **session-level mutex** (`withSessionLock`) to prevent read-modify-write races during concurrent updates. Atomically writes state via temp file + rename. **Manages cycle snapshots**: `createSnapshot()` (at cycle boundaries), `restoreSnapshot()` (restore to prior state), `listSnapshots()` (available snapshots), `deleteSnapshotsAfter()` (cleanup post-rollback). **`updateAgent()` searches in reverse** — `.slice().reverse().find()` — so if multiple entries share the same `agentId`, the most-recently-appended one is updated; callers that push duplicate IDs will silently update the wrong record if they add rather than mutate. **`restoreSnapshot()` clears `tmuxSessionName` and `tmuxWindowId`** from the restored state — the existing tmux session/window is NOT killed, just orphaned; `resumeSession()` must be called afterward to re-bind the session to tmux.
- **tmux.ts** — tmux CLI wrapper: session/window/pane creation and lifecycle. **Session defaults**: every `createSession()` calls `configureSessionDefaults()` which sets `pane-border-status top`, disables `allow-rename`/`automatic-rename` (prevents tmux from clobbering pane titles we set), and installs `after-kill-pane`/`pane-exited` hooks that auto-retile to `even-horizontal`. **Also renames the window to `"main"`** after disabling automatic-rename — necessary because `automatic-rename` may have already changed the window name to the cwd basename before we disable it; if another session has a window with that same basename, `tmux set-option -t "session:basename"` silently targets the wrong session. **Pane creation**: `createPane()` targets the first/last existing pane (not window) to guarantee absolute left/right placement — using the window as target splits relative to the active pane. **Home session support**: `findHomeSession()` locates non-sisyphus sessions in same cwd (skips `ssyph_`-prefixed names); `switchAttachedClients()` moves attached clients to home session before destroying sisyphus session. **Global helpers**: `getGlobalOption()` (reads via `show-option -gv`, swallows errors → returns null), `setGlobalOption()`, `listAllSessions()`, `listAllPanes()` — used by `status-bar.ts` for whole-server status rendering. `getGlobalOption` never throws; `setGlobalOption` propagates errors (tmux errors are fatal). **Pane styling**: `setPaneStyle()` stores metadata as per-pane user vars (`@pane_role`, `@pane_session`, `@pane_cycle`, `@pane_mode`, `@pane_color`) and `pane-border-format` — all set with `-p` (per-pane flag) so each pane has its own label format; window-level `pane-border-style` uses `#{@pane_color}` which resolves per-pane at render time (last-write-wins for the window option itself). `updatePaneMeta()` updates individual vars without rebuilding the full format string (used by auto-naming).
- **companion.ts** — XP/level/mood/achievement system persisted to `~/.sisyphus/companion.json` via its own temp+rename (not `state.ts` mutex). `loadCompanion()` applies forward-compat field fills on read. `computeMood()` with no `signals` arg returns hour-based mood only (existential 02–06, sleepy 22–02, zen otherwise) — signals arg is required for crash/session-length-aware scoring. `onSessionComplete()` returns an array of newly unlocked `AchievementId`s (empty if none) — callers must check this to trigger achievement commentary. XP formula: strength×80 + endurance/3.6M×15 + wisdom×40 + patience×5.
- **companion-commentary.ts** — Haiku-generated one-liners for lifecycle events. 5-minute cooldown on 401/403 (module-level `disabledUntil`). `shouldGenerateCommentary()` probabilities: `session-start`/`session-complete`/`level-up`/`achievement`/`late-night` → always; `cycle-boundary`/`idle-wake` → 50%; `agent-crash` → 30%. All non-zero — no event type returns null unconditionally. Time-of-day personality: chipper 06–10, professional 10–17, reflective 17–22, dry-humor 22–02, delirious 02–06.
- **colors.ts** — Color cycling for panes: orchestrator always yellow, agents rotate deterministically.

## Key Patterns

**Respawn guard race prevention**: `respawningSessions` tracks sessions in yield→respawn transition. Pane monitor skips empty-window detection for guarded sessions (window may be temporarily empty between killing orchestrator and spawning new one). Guard added on yield/respawn entry, cleared on completion or error — but **not cleared if `onAllAgentsDone()` returns early** (e.g., session not `active` at entry). An early-return without clearing leaves the session stuck in `between-cycles` forever.

**Active time accumulation**: Pane monitor maintains in-memory timers for each session (session, agents, cycles). On each poll, increments timers by elapsed time if panes are alive (using sleep-aware calculation). `flushTimers()` persists deltas to state file. Called before session completion/kill/rollback.

**Session completion & client switching**: When completing or killing a session via `handleComplete()`/`handleKill()`, `switchToHomeSession()` moves any attached tmux clients to a home session (if available in same cwd) before destroying the sisyphus session. Prevents users from being left in a killed session.

**Session creation**: Creates tmux session internally with deterministic naming (`sisyphus-{name}` if provided, else `sisyphus-{sessionId[:8]}`). Validates custom names against alphanumeric/hyphen/underscore pattern. Auto-generates human-readable session names from task via Haiku (fire-and-forget, failures are silent). Kills initial pane spawned by `tmux new-session` after orchestrator spawns its own.

**Agent spawning**: Increments counter, resolves agent type frontmatter (model, color, effort, interactive), detects provider (Anthropic Claude or OpenAI Codex), creates Claude plugin with conditional hooks per agent type, spawns pane.

**Session resumption**: Reuses existing tmux session if available, else creates fresh one. Uses `listPanes()` to detect live agent panes and mark lost agents. Reuses windowId from state if tmux session still exists. Kills initial pane after orchestrator spawns.

**Window reopening**: `reopenWindow()` recreates tmux window/session if killed (e.g., after orchestrator yields with no running agents), reuses existing IDs if available.

**Respawn on all-agents-done**: When all agents finish, `onAllAgentsDone()` adds session to `pendingRespawns`, creates snapshot of current cycle state, ensures tmux session/window exist (recreates if necessary), then respawns orchestrator via `setImmediate`. Uses `pendingRespawns` + `orchestratorDone` guards to prevent concurrent respawns. `orchestratorDone` is checked both in-memory AND by reading `lastCycle.completedAt` from persisted state — the state fallback handles daemon restart between orchestrator yield and agents finishing. `respawningSessions` guard (via respawn-guard.ts) prevents pane monitor from pausing during yield→respawn window.

**Session recovery on startup**: Registry loads; `active`/`paused` sessions recover. Detects stuck sessions (all agents done, orchestrator pane gone) and immediately triggers respawn.

**Cycle snapshots**: At cycle boundaries, `state.createSnapshot()` captures full session state. `handleRollback()` restores to prior snapshot, killing running agents, removing post-rollback snapshots. `handleRollback()` does **not** restart the session — leaves it in restored state; caller must issue `resumeSession()` explicitly.

**Orchestrator crash branching**: `handlePaneExited(role='orchestrator')` forks on agent count: if agents are running → set guard + wait; if all agents already done → trigger `onAllAgentsDone()`; if `agents.length === 0` → pause session (no respawn target). Completed agent panes are killed **lazily** inside `onAllAgentsDone`'s `setImmediate` callback after the new orchestrator spawns, not at submit time.

**Kill-agent ordering**: `handleKillAgent()` unregisters the pane from `pane-registry` before killing the tmux pane. This prevents pane monitor from seeing an unexpected exit and triggering an unintended respawn for a user-initiated kill.

**Companion write races**: `companion.ts` uses its own load/save (not `state.ts` mutex), so concurrent writes are possible. Async commentary callbacks always call `loadCompanion()` fresh immediately before writing `lastCommentary` to minimize the stomp window — never capture a companion reference across an `await`. Multiple commentary chains (session-complete + level-up + achievement) can overlap; the last writer wins on `lastCommentary`, which is acceptable since each chain writes a different event type.

**Prompts**: System + user prompts written to `prompts/` subdir. Passed via Claude flags to avoid tmux quoting issues.

## Constraints

- Never write state directly — use `state.ts` atomic writes
- Always unregister panes from pane-registry and pane-monitor when killing/completing
- Prompt files must be on disk in `prompts/` subdir
- tmux errors are fatal — propagate, don't swallow
- Session recovery must detect and recover stuck sessions immediately on startup
- Agent type frontmatter must be valid YAML; gracefully fall back if missing or malformed
- Snapshots created at cycle boundaries; rollback validates snapshot existence
- Kill initial pane after orchestrator spawns (tmux creates it but orchestrator needs its own)
- Custom session names must be alphanumeric, hyphens, underscores only
- Session names containing tmux format special characters (`#`, `{`, `}`) break `status-bar.ts` format strings silently — no escaping is applied
- Interactive agents (interactive: true) skip the Stop hook requirement; all others require submit hook
- Haiku-based naming/summarization is optional (fire-and-forget); auth failures silence feature temporarily with 5-minute cooldown
- Active time must be flushed to state before session completion/kill/rollback (prevents loss of accumulated time data)
- Respawn guard must be cleared on respawn completion or error; **`onAllAgentsDone()` has two early-return paths with different guard behavior**: (1) if `orchestratorDone` is not set yet (waiting for orchestrator to yield), returns without clearing — guard stays, session correctly shows `between-cycles`; (2) if `session.status !== 'active'`, deletes the guard before returning — but no respawn fires, leaving the session unpaused from `between-cycles` but stuck with no orchestrator. Path (2) is the dangerous one: recover by kill + restart.
- `orchestratorDone` IS cleared by `resumeSession()`, but that doesn't rescue the stuck state from path (2) above — `onAllAgentsDone()` already returned without respawning, and `resumeSession()` doesn't re-trigger the check. The `lastCycle.completedAt` fallback only catches clean orchestrator yields — if the orchestrator crashed (no `completedAt` written), the fallback finds nothing and respawn stays blocked even after daemon restart; recovery is kill + restart.
- Call `recomputeDots()` (best-effort try/catch) after any handler that changes session phase — omitting it leaves the dashboard stale until the next pane-monitor poll
- Companion writes use own `loadCompanion()`/`saveCompanion()` — never route through `state.ts`. Async commentary callbacks must reload fresh before saving (never close over a captured companion reference across an await)
- `onSessionComplete()` return value (new achievement IDs) must be inspected — achievement commentary only fires if the array is non-empty; the companion is already mutated when the function returns
