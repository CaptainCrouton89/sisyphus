# CLI Commands

Each file exports a command registration function following this pattern:

```ts
export function register{Command}(program: Command): void {
  program.command('{command}').action(async (args, opts) => { ... });
}
```

## Conventions

- **Registration**: Each command file exports `register{Command}()` called in `index.ts`
- **Communication**: Commands use `sendRequest()` to send protocol requests via Unix socket
- **Error handling**: Throw on invalid args or daemon errors; exit with code 1 and error message
- **Session lookup**: Commands requiring session ID use `process.env.SISYPHUS_SESSION_ID` or error if missing
- **Output**: Use `console.log()` for success; `console.error()` for errors
- **Validation**: Early validation of required args; avoid side effects before confirming validity
- **Stdin support**: Use `readStdin()` for optional input piping (e.g., `spawn`, `yield`)

## Key Constraints

- **init.ts** — Creates `.sisyphus/config.json` and optional `orchestrator.md` template. Option: `--orchestrator` (create custom prompt). Idempotent (logs and exits if config exists).
- **start.ts** — Requires tmux (checks `TMUX` env var; skip with `--no-tmux-check`). Options: `--context` (background info), `--name` (session label). Respects `SISYPHUS_CWD` env var (falls back to `process.cwd()`). Auto-launches dashboard in current tmux session (only if not already open). Sets `@sisyphus_cwd` tmux option.
- **continue.ts** — Clears roadmap and reactivates completed session (stays in current cycle); requires `SISYPHUS_SESSION_ID` env var.
- **resume.ts** — Takes session ID as **positional argument** (not env var). Optional second arg for additional orchestrator instructions. Returns tmux session name for attaching.
- **submit.ts** — Submits agent work report. Requires `SISYPHUS_SESSION_ID` and `SISYPHUS_AGENT_ID` env vars.
- **spawn.ts** — Orchestrator-only; requires `SISYPHUS_SESSION_ID`. Instruction from positional arg, `--instruction` flag, or stdin. Options: `--agent-type` (default: `worker`), `--name` (required). **`--repo <name>`** specifies subdirectory for multi-repo workflows (directory name only, no paths). If omitted, agents work in the session root.
- **yield.ts** — Orchestrator-only; requires `SISYPHUS_SESSION_ID` environment variable.
- **rollback.ts** — Arguments: `<sessionId> <cycle>` (cycle must be positive integer). Pauses session; use `resume` to respawn.
- **dashboard.ts** — Checks if dashboard window exists before launching (prevents duplicates). Launches in current tmux session via TUI binary.
- **list.ts** — Lists sessions for current project (default) or all projects (`--all`). Color-codes status: green=active, yellow=paused, cyan=completed. Respects `SISYPHUS_CWD` for project filtering; override with `--cwd <path>`.
- **doctor.ts** — Platform-aware health checks; verifies dependencies and tmux/daemon setup.
- **getting-started.ts** — Interactive guide; checks tmux install/status and displays workflow guidance.
- **setup.ts** — One-time setup: runs `runOnboarding()` (tmux, iTerm, nvim, `/begin` command), then installs daemon, then writes tmux keybinds. Daemon install failure falls back to `isInstalled()` — if daemon is already running the check still shows green rather than failing. Prints instruction to add `#{@sisyphus_status}` to `status-right`. `runOnboarding()` side effects: may auto-install LazyVim starter config to `~/.config/nvim/` (only if nvim present and no existing config); checks iTerm right option key per profile and warns if not set to `Esc+` (does not fix automatically).
- **status.ts** — Session ID resolution: positional arg → `SISYPHUS_SESSION_ID` → `undefined` (daemon picks current session). `inferOrchestratorPhase()` is a **local heuristic** (not daemon-provided): if last cycle has no `completedAt` and elapsed < 5000ms or no agents spawned → `'planning'`, else → `'spawning'`; if cycle has `completedAt` and agents from that cycle are still running → `'waiting on {ids}'`; if cycle has `completedAt` and no running agents → `'starting'` (daemon is about to respawn orchestrator). Only shown on the last cycle of an active session. `computeLastActivity()` scans all cycle start/end timestamps + agent timestamps + report timestamps — there is no single `lastActivity` field in state. Roadmap is **always** printed when present — not gated on `--verbose` (despite the flag's help text saying "roadmap"). `--verbose` adds: agent instructions, cycle logs (capped at 20 lines), live tmux pane output (orchestrator: 50 lines if last cycle has no `completedAt` AND has `paneId`; each running agent: 30 lines), and `session.completionReport` if present. Pane output XML tags carry attributes: `<orchestrator-pane-output lines="50">` and `<agent-pane-output agent="{id}" name="{name}" lines="30">` — structured for machine/Claude consumption. The `cwd` resolved locally (from `SISYPHUS_CWD` or `process.cwd()`) is passed only for daemon request routing; filesystem reads (`readRoadmap`, `readCycleLog`) use `session.cwd` from the returned state.
- **tmux-status.ts** — Reads the daemon-pre-rendered `@sisyphus_status` **global** tmux option (`show-option -gv`, not per-session `-t`) and writes it to stdout unchanged. No rendering logic — daemon owns the format. Silent on error or unset. This is the preferred `status-right` source (`#{@sisyphus_status}`); `tmux-sessions` is the fallback that does local rendering.
- **tmux-sessions.ts** — Outputs a tmux `status-right` fragment via `process.stdout.write()` (no trailing newline — `console.log` would break tmux format string parsing). **CWD scoping reads `~/.sisyphus/sessions-manifest.json`** (daemon-written), not tmux options: looks up the current session by name to get its `cwd`, then filters all manifest entries to those sharing that `cwd`. **Silent exit conditions** (emits nothing): manifest file missing, current session not found in manifest, or fewer than 2 entries share the same `cwd` — intentional, avoids cluttering single-session status bars. Display name strips `ssyph_{id}_` (two segments: `/^ssyph_[^_]+_/`) — e.g. `ssyph_abc123_my-task` → `my-task`. The session matching `tmux display-message -p '#{session_name}'` renders bold+light (`#e2d9c6`); all others render dim (`#5e584e`) — purely name comparison, no flag. Phase → icon/color mapping is local (`DOT_MAP`); values come from `ManifestEntry.phase`, not from per-session tmux options.
- **review.ts** — Registers two commands (`requirements`, `design`) via shared `runReviewTui()`. Does **not** use `sendRequest()` — no daemon involvement. Binary resolved via `import.meta.dirname/{binaryName}` (sibling to CLI in dist/): `requirements` → `review.js`, `design` → `design.js`. File resolution order: positional arg → session ID (opt or `SISYPHUS_SESSION_ID` env) → most-recent session scan (reverses directory listing under `.sisyphus/sessions/`, first match wins). Session-resolved path is `.sisyphus/sessions/{id}/context/{filename}` — note the `context/` subdirectory (not session root). **`--wait` implies `--window`** (undocumented in help): `useWindow = opts.window || opts.wait`. `--window`/`--wait` mode uses `tmux new-window` + `tmux wait-for` with a random per-invocation channel (`review-{8hex}`) to avoid collisions from parallel reviews. `--wait` reads feedback from `dirname(targetPath)/{feedbackFilename}` (same `context/` dir as the artifact, written by TUI), prints to stdout, then **deletes the file** — caller receives feedback via stdout only. **History events**: `--wait` with a session ID (opt or env) emits `review-started` to `~/.sisyphus/history/{id}/events.jsonl` before blocking, then `review-completed` after unblocking. Item counts for `review-completed` are extracted from the artifact JSON post-review — `requirements.json` flattens `groups[].requirements[]`, `design.json` flattens `sections[].items[]`; an item counts as reviewed if `reviewAction` is truthy **or** `status === 'approved'`. Duration is read from `meta.reviewStartedAt`/`meta.reviewCompletedAt` written into the artifact by the TUI (not tracked by this command independently). **`--schema` / `--annotated`** short-circuit before any file resolution — print to stdout and exit immediately; no tmux or session required. The schemas and annotated writing guides are **hardcoded constants in `review.ts`** (not external files); update them there if the artifact format changes. **Schema divergence between commands**: requirements `reviewAction` is `'approve' | 'comment' | null`; design `reviewAction` is `'agree' | 'pick-alt' | 'comment' | null` — both satisfy the `reviewAction` truthy check for item counting, but code consuming design artifacts must handle `pick-alt` + `selectedAlternative`. Design items also carry an optional `decision` object (`proposal`, `alternatives[]`, `lenses{}`) that has no requirements equivalent.
- **present.ts** — Does **not** LCS-diff or wrap edits in HTML comment tags (parent CLAUDE.md description is stale). `--wait` mode: opens a `tmux split-window -h -l 50%` (pane, not window), blocks via `tmux wait-for`, reads back the temp `.ansi` file verbatim, writes it to stdout, then cleans up. Two temp files per invocation: `sisyphus-present-{hex}.ansi` (rendered content) + `sisyphus-present-{hex}-init.lua` (baleia Lua init). **`--width` default `120` is a sentinel** — actual render width auto-calculates as `floor(pane_width/2) - 10` (half-pane minus tmux border + nvim sign column); explicit `--width` overrides this. **Baleia**: nvim loaded with `-S initPath`; the init script uses `vim.defer_fn(..., 50)` because the buffer isn't attached at startup — removing the defer breaks ANSI rendering. **`--no-wait`**: temp files deliberately not cleaned up (nvim still holds them open).
- **history.ts** — Reads directly from `~/.sisyphus/history/{sessionId}/` — no daemon involved. Two files per session: `session.json` (summary snapshot written at session end) and `events.jsonl` (NDJSON append log). Three modes dispatched by arg/flag presence in priority order: `--stats` → positional `[session]` → bare list; `--stats` with a positional arg silently ignores the arg. Session lookup for detail view: exact ID match → ID prefix match → exact name match → name substring match. **`findSession` name matching is case-sensitive** (no `.toLowerCase()`); `--search` is case-insensitive — inconsistency by design. `--search` scans `summary.task`, `summary.name`, and all `summary.messages[].content` (messages are denormalized into the summary at session end — not re-read from the event log). `review-completed` events in the detail view are loaded from the event log, not the summary — they appear regardless of whether the summary captured them. `--since` format is `{n}{d|h|m|w}` only; other formats exit 1. **`--events` flag**: switches the positional-arg detail view to raw event timeline; `--json` + `--events` returns a JSON array of `HistoryEvent` objects (not the summary object that `--json` alone returns). **`efficiency` in `--stats`**: reads `summary.efficiency` first; missing on older sessions falls back to `activeMs / wallClockMs` — absent from stats if both are unavailable. **Temporal patterns** in `--stats` only render when `sessions.length >= 5`; silently omitted below that threshold.

## Key Interactions

- `init.ts` → (optional) run before `start.ts` to create project config
- `start.ts` → creates new session, spawns orchestrator, auto-launches dashboard
- `spawn.ts` → orchestrator spawns agents into panes
- **`continue.ts` vs `resume.ts`** — Both reactivate sessions; use `continue` for same direction, `resume` for new instructions
- `submit.ts`, `yield.ts`, `complete.ts` — Lifecycle commands; require active session
- `status.ts`, `dashboard.ts` — Query/monitor; read-only
- `list.ts` — Lists all sessions (project-scoped or global); read-only
- `rollback.ts`, `kill.ts` — Session management; don't require active session
- `doctor.ts`, `getting-started.ts` — Standalone utilities; no session dependency
