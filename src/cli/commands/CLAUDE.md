# CLI Commands

Each file exports a command registration function following this pattern:

```ts
export function register{Command}(program: Command): void {
  program.command('{command}').action(async (args, opts) => { ... });
}
```

## Conventions

- **Registration**: Each command file exports `register{Command}()` called in `index.ts`
- **Communication**: Commands use `sendRequest()` to send protocol requests via Unix socket
- **Error handling**: Throw on invalid args or daemon errors; exit with code 1 and error message
- **Session lookup**: Commands requiring session ID use `process.env.SISYPHUS_SESSION_ID` or error if missing
- **Output**: Use `console.log()` for success; `console.error()` for errors
- **Validation**: Early validation of required args; avoid side effects before confirming validity
- **Stdin support**: Use `readStdin()` for optional input piping (e.g., `spawn`, `yield`)

## Key Constraints

- **init.ts** тАФ Creates `.sisyphus/config.json` and optional `orchestrator.md` template. Option: `--orchestrator` (create custom prompt). Idempotent (logs and exits if config exists).
- **start.ts** тАФ Requires tmux (checks `TMUX` env var; skip with `--no-tmux-check`). Options: `--context` (background info), `--name` (session label). Respects `SISYPHUS_CWD` env var (falls back to `process.cwd()`). Auto-launches dashboard in current tmux session (only if not already open). Sets `@sisyphus_cwd` tmux option.
- **continue.ts** тАФ Clears roadmap and reactivates completed session (stays in current cycle); requires `SISYPHUS_SESSION_ID` env var.
- **resume.ts** тАФ Takes session ID as **positional argument** (not env var). Optional second arg for additional orchestrator instructions. Returns tmux session name for attaching.
- **submit.ts** тАФ Submits agent work report. Requires `SISYPHUS_SESSION_ID` and `SISYPHUS_AGENT_ID` env vars.
- **spawn.ts** тАФ Orchestrator-only; requires `SISYPHUS_SESSION_ID`. Instruction from positional arg, `--instruction` flag, or stdin. Options: `--agent-type` (default: `worker`), `--name` (required). **`--repo <name>`** specifies subdirectory for multi-repo workflows (directory name only, no paths). If omitted, agents work in the session root.
- **yield.ts** тАФ Orchestrator-only; requires `SISYPHUS_SESSION_ID` environment variable.
- **rollback.ts** тАФ Arguments: `<sessionId> <cycle>` (cycle must be positive integer). Pauses session; use `resume` to respawn.
- **dashboard.ts** тАФ Checks if dashboard window exists before launching (prevents duplicates). Launches in current tmux session via TUI binary.
- **list.ts** тАФ Lists sessions for current project (default) or all projects (`--all`). Color-codes status: green=active, yellow=paused, cyan=completed. Respects `SISYPHUS_CWD` for project filtering; override with `--cwd <path>`.
- **doctor.ts** тАФ No daemon contact (all checks are local filesystem/process only). Always exits 0 regardless of failures тАФ not CI-usable as a health gate. **iTerm option key** is the only check that can be absent from output: `checkItermRightOptionKey()` returns `null` (omitted) on non-macOS, when not running iTerm2, or when the plist is unreadable. **Tmux keybind null ambiguity**: `getExistingBinding()` returns `null` both when the key is unbound AND when tmux isn't running тАФ disambiguated by falling back to checking `sisyphusTmuxConfPath()` exists (тЖТ `warn` instead of `fail`). **Daemon check platform split**: macOS uses `isInstalled()` (checks launchd plist); Linux uses PID file existence (not plist).
- **getting-started.ts** тАФ Interactive guide; checks tmux install/status and displays workflow guidance.
- **setup.ts** тАФ One-time setup: runs `runOnboarding()` (tmux, iTerm, nvim, `/begin` command), then installs daemon, then writes tmux keybinds. Daemon install failure falls back to `isInstalled()` тАФ if daemon is already running the check still shows green rather than failing. Prints instruction to add `#{@sisyphus_status}` to `status-right`. `runOnboarding()` side effects: may auto-install LazyVim starter config to `~/.config/nvim/` (only if nvim present and no existing config); checks iTerm right option key per profile and warns if not set to `Esc+` (does not fix automatically).
- **status.ts** тАФ Session ID resolution: positional arg тЖТ `SISYPHUS_SESSION_ID` тЖТ `undefined` (daemon picks current session). `inferOrchestratorPhase()` is a **local heuristic** (not daemon-provided): if last cycle has no `completedAt` and elapsed < 5000ms or no agents spawned тЖТ `'planning'`, else тЖТ `'spawning'`; if cycle has `completedAt` and agents from that cycle are still running тЖТ `'waiting on {ids}'`; if cycle has `completedAt` and no running agents тЖТ `'starting'` (daemon is about to respawn orchestrator). Only shown on the last cycle of an active session. `computeLastActivity()` scans all cycle start/end timestamps + agent timestamps + report timestamps тАФ there is no single `lastActivity` field in state. Roadmap is **always** printed when present тАФ not gated on `--verbose` (despite the flag's help text saying "roadmap"). `--verbose` adds: agent instructions, cycle logs (capped at 20 lines), live tmux pane output (orchestrator: 50 lines if last cycle has no `completedAt` AND has `paneId`; each running agent: 30 lines), and `session.completionReport` if present. Pane output XML tags carry attributes: `<orchestrator-pane-output lines="50">` and `<agent-pane-output agent="{id}" name="{name}" lines="30">` тАФ structured for machine/Claude consumption. The `cwd` resolved locally (from `SISYPHUS_CWD` or `process.cwd()`) is passed only for daemon request routing; filesystem reads (`readRoadmap`, `readCycleLog`) use `session.cwd` from the returned state.
- **tmux-status.ts** тАФ Reads the daemon-pre-rendered `@sisyphus_status` **global** tmux option (`show-option -gv`, not per-session `-t`) and writes it to stdout unchanged. No rendering logic тАФ daemon owns the format. Silent on error or unset. This is the preferred `status-right` source (`#{@sisyphus_status}`); `tmux-sessions` is the fallback that does local rendering.
- **tmux-sessions.ts** тАФ Outputs a tmux `status-right` fragment via `process.stdout.write()` (no trailing newline тАФ `console.log` would break tmux format string parsing). **CWD scoping reads `~/.sisyphus/sessions-manifest.json`** (daemon-written), not tmux options: looks up the current session by name to get its `cwd`, then filters all manifest entries to those sharing that `cwd`. **Silent exit conditions** (emits nothing): manifest file missing, current session not found in manifest, or fewer than 2 entries share the same `cwd` тАФ intentional, avoids cluttering single-session status bars. Display name strips `ssyph_{id}_` (two segments: `/^ssyph_[^_]+_/`) тАФ e.g. `ssyph_abc123_my-task` тЖТ `my-task`. The session matching `tmux display-message -p '#{session_name}'` renders bold+light (`#e2d9c6`); all others render dim (`#5e584e`) тАФ purely name comparison, no flag. Phase тЖТ icon/color mapping is local (`DOT_MAP`); values come from `ManifestEntry.phase`, not from per-session tmux options.
- **review.ts** тАФ Registers two commands (`requirements`, `design`) via shared `runReviewTui()`. Does **not** use `sendRequest()` тАФ no daemon involvement. Binary resolved via `import.meta.dirname/{binaryName}` (sibling to CLI in dist/): `requirements` тЖТ `review.js`, `design` тЖТ `design.js`. File resolution order: positional arg тЖТ session ID (opt or `SISYPHUS_SESSION_ID` env) тЖТ most-recent session scan (reverses directory listing under `.sisyphus/sessions/`, first match wins). Session-resolved path is `.sisyphus/sessions/{id}/context/{filename}` тАФ note the `context/` subdirectory (not session root). **`--wait` implies `--window`** (undocumented in help): `useWindow = opts.window || opts.wait`. `--window`/`--wait` mode uses `tmux new-window` + `tmux wait-for` with a random per-invocation channel (`review-{8hex}`) to avoid collisions from parallel reviews. `--wait` reads feedback from `dirname(targetPath)/{feedbackFilename}` (same `context/` dir as the artifact, written by TUI), prints to stdout, then **deletes the file** тАФ caller receives feedback via stdout only. **History events**: `--wait` with a session ID (opt or env) emits `review-started` to `~/.sisyphus/history/{id}/events.jsonl` before blocking, then `review-completed` after unblocking. Item counts for `review-completed` are extracted from the artifact JSON post-review тАФ `requirements.json` flattens `groups[].requirements[]`, `design.json` flattens `sections[].items[]`; an item counts as reviewed if `reviewAction` is truthy **or** `status === 'approved'`. Duration is read from `meta.reviewStartedAt`/`meta.reviewCompletedAt` written into the artifact by the TUI (not tracked by this command independently). **`--schema` / `--annotated`** short-circuit before any file resolution тАФ print to stdout and exit immediately; no tmux or session required. The schemas and annotated writing guides are **hardcoded constants in `review.ts`** (not external files); update them there if the artifact format changes. **Schema divergence between commands**: requirements `reviewAction` is `'approve' | 'comment' | null`; design `reviewAction` is `'agree' | 'pick-alt' | 'comment' | null` тАФ both satisfy the `reviewAction` truthy check for item counting, but code consuming design artifacts must handle `pick-alt` + `selectedAlternative`. Design items also carry an optional `decision` object (`proposal`, `alternatives[]`, `lenses{}`) that has no requirements equivalent.
- **present.ts** тАФ Does **not** LCS-diff or wrap edits in HTML comment tags (parent CLAUDE.md description is stale). `--wait` mode: opens a `tmux split-window -h -l 50%` (pane, not window), blocks via `tmux wait-for`, reads back the temp `.ansi` file verbatim, writes it to stdout, then cleans up. Two temp files per invocation: `sisyphus-present-{hex}.ansi` (rendered content) + `sisyphus-present-{hex}-init.lua` (baleia Lua init). **`--width` default `120` is a sentinel** тАФ actual render width auto-calculates as `floor(pane_width/2) - 10` (half-pane minus tmux border + nvim sign column); explicit `--width` overrides this. **Baleia**: nvim loaded with `-S initPath`; the init script uses `vim.defer_fn(..., 50)` because the buffer isn't attached at startup тАФ removing the defer breaks ANSI rendering. **`--no-wait`**: temp files deliberately not cleaned up (nvim still holds them open).
- **history.ts** тАФ Reads directly from `~/.sisyphus/history/{sessionId}/` тАФ no daemon involved. Two files per session: `session.json` (summary snapshot written at session end) and `events.jsonl` (NDJSON append log). Three modes dispatched by arg/flag presence in priority order: `--stats` тЖТ positional `[session]` тЖТ bare list; `--stats` with a positional arg silently ignores the arg. Session lookup for detail view: exact ID match тЖТ ID prefix match тЖТ exact name match тЖТ name substring match. **`findSession` name matching is case-sensitive** (no `.toLowerCase()`); `--search` is case-insensitive тАФ inconsistency by design. `--search` scans `summary.task`, `summary.name`, and all `summary.messages[].content` (messages are denormalized into the summary at session end тАФ not re-read from the event log). `review-completed` events in the detail view are loaded from the event log, not the summary тАФ they appear regardless of whether the summary captured them. `--since` format is `{n}{d|h|m|w}` only; other formats exit 1. **`--events` flag**: switches the positional-arg detail view to raw event timeline; `--json` + `--events` returns a JSON array of `HistoryEvent` objects (not the summary object that `--json` alone returns). **`efficiency` in `--stats`**: reads `summary.efficiency` first; missing on older sessions falls back to `activeMs / wallClockMs` тАФ absent from stats if both are unavailable. **Temporal patterns** in `--stats` only render when `sessions.length >= 5`; silently omitted below that threshold.

## Key Interactions

- `init.ts` тЖТ (optional) run before `start.ts` to create project config
- `start.ts` тЖТ creates new session, spawns orchestrator, auto-launches dashboard
- `spawn.ts` тЖТ orchestrator spawns agents into panes
- **`continue.ts` vs `resume.ts`** тАФ Both reactivate sessions; use `continue` for same direction, `resume` for new instructions
- `submit.ts`, `yield.ts`, `complete.ts` тАФ Lifecycle commands; require active session
- `status.ts`, `dashboard.ts` тАФ Query/monitor; read-only
- `list.ts` тАФ Lists all sessions (project-scoped or global); read-only
- `rollback.ts`, `kill.ts` тАФ Session management; don't require active session
- `doctor.ts`, `getting-started.ts` тАФ Standalone utilities; no session dependency
