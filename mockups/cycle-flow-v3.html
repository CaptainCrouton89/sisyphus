DESIGN MODEL
Three-Phase Sliding Window
The lifecycle is a sequence of phases: orchestrator → agents → yield → orchestrator → agents → … We show exactly 3 phases at a time: the one before, the current one, and what's coming next. The window slides forward as the session progresses.
visual language
Visual Language Brightness: ╭──────────╮ dim/faded = completed, in the past ╭──────────╮ bright/bold = active right now ╌ ╌ ╌ ╌ ╌ ╌ dotted/dashed = planned, hasn't happened yet Background tints (ANSI bg colors, dark-tinted): ● C2 impl running ← orchestrator: yellow bg ▶ a-004 backend 12m ← agent: its assigned color bg ▶ a-005 migrate 8m ← agent: green bg yield: "Run compliance…" ← yield prompt: gray bg Status icons: ● orchestrator running ▶ agent running ○ orchestrator done ✓ agent completed ✕ killed ! crashed ? lost Mode colors (on orchestrator nodes): plan impl strat valid
KEY INSIGHT
Background tints replace box borders as the primary grouping signal. A tinted row reads as "this is a node" without needing ╭──╮ everywhere. Borders become optional structural lines. This is more compact and avoids box-drawing alignment problems.

BG COLOR IN PRACTICE
In a real terminal, bg tints use ANSI 256-color or truecolor escapes: \x1b[48;2;R;G;B m. The tint values are dark enough to keep text legible on the dark terminal background. Each agent's assigned color (blue, green, magenta, etc.) maps to a dim version: rgb(30, 40, 70) for blue bg, rgb(30, 50, 30) for green bg, etc. Only needs ~6 bg colors total.

SINGLE AGENT BEHAVIOR
With only 1 agent, the box expands to full width and shows the instruction text inside the bg-tinted area. More useful than a tiny box — you can see what it's doing.
DASHED LINES: IMPLEMENTATION
The "planned/future" dashed outline uses ┊ for verticals and spaced ╌ for horizontals (┊ ╌ ╌ ╌ ┊ with spaces between). The spaces are intentional — they read as "dotted" rather than trying to make ╌╌╌ connect (which fails). This is a feature, not a bug.
