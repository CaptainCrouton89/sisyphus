# Companion Roadmap

## Shipped
- Intensity-based mood faces (3 tiers driven by mood score)
- ☯ zen prefix
- Session history tracking (~/.sisyphus/history/)
- `sisyphus history` CLI
- Spinner verbs (cycle each orchestrator cycle, active sessions only)
- Idle hobbies (display when no sessions active, replace spinner verb slot)

## In progress

### Sentiment extraction
Each session gets a one-sentence "sentiment read" generated by Haiku at completion. Captures how the user felt about the work — not what they did, but the emotional register. Stored in session history alongside task/messages.

Feed the last 3 session sentiments into the companion's commentary prompt so it can reference the user's recent emotional arc. A companion that knows "you've been frustrated with architecture debt all week but feeling better about the refactor" is fundamentally different from one that just knows session counts.

Open questions:
- Do we generate sentiment at completion only, or also mid-session (e.g., after each user message)?
- How much task context does Haiku need to read the sentiment? Full task + messages, or just the task string?
- Storage: add `sentiment` field to session.json in history, or separate file?

### Commentary tone overhaul
Rewrite the Haiku system prompts in companion-commentary.ts to sound human per the sounding-human guide. Keep time-of-day variation but make each personality feel like an actual person, not a character archetype. The companion is a philosophical, slightly ironic sidekick — self-aware, slightly tragic, sometimes insightful.

Current state: personalities are named (chipper, professional, reflective, dry-humor, delirious) but the prompts produce generic output. Need to inject more voice, more specificity, less bot.

### Companion memory
The companion should accumulate observations about the user over time — not just stat counters but qualitative understanding. This feeds into commentary generation.

Possible approach: a small structured file (~/.sisyphus/companion-memory.json) with:
- Recent session sentiments (rolling window)
- Repo impressions (beyond visits/crashes — actual characterization)
- User patterns ("works late", "prefers quality over speed", "gets frustrated with tech debt")
- Notable moments ("first 10-agent session", "longest streak")

### Direct conversation with companion
Users should be able to talk to the companion directly — not just read its status bar quips. Could be:
- `sisyphus companion chat` — a short exchange
- A TUI overlay mode
- Integration into the orchestrator prompt so the companion can comment during sessions

This is the furthest out. Needs design for what the interaction model looks like and what the companion can actually do in response.

## Future ideas
- Companion reacts to git state (big merge, conflict resolution, force push)
- Repo-specific personality shifts (companion is nervous in repos with high crash history)
- Companion "remembers" past sessions and references them in commentary
- Seasonal/calendar awareness (holidays, weekends, late nights)
- Multiple companion personas/skins
